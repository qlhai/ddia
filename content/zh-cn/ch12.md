在[第十一章](/zh-cn/ch11#ch_batch)中，我们探讨了批处理（批处理（batch processing））：一类以若干输入文件为源、生成若干输出文件的技术。其输出属*派生数据*（derived data），即必要时可通过重跑批处理流程重新生成之数据集。我们已见此简明而强效之思想如何用于构建搜索索引、推荐系统、分析报表等。

然第十一章始终隐含一大前提：输入数据为**有界**（bounded）——即规模已知且有限，故批处理器可判定输入读取完毕之时点。例如，MapReduce 中居核心地位之排序操作，须待全部输入读毕，方得始产输出：因末条输入记录或恰为键值最小者，须列为首条输出记录；故不可提前输出。

现实中，大量数据实为**无界**（unbounded）：数据随时间渐次抵达——用户昨日、今日产数据，明日亦将如是。除非业务终止，此过程永无休止；故该数据集在任何实质意义上皆永无“完成”之日[^1]。因此，批处理器须人为将数据切分为固定时长之块：例如，每日终了处理当日全部数据，或每小时终了处理当小时全部数据。

日级批处理之弊，在于输入变动需一日后方显于输出，对诸多急切用户而言，延迟过甚。欲减此延迟，可提高处理频度——譬如每秒终了处理一秒之数据——甚或彻底摒弃固定时窗，转为持续处理，即事件一至即刻处理。此即*流处理*（流处理（stream processing））之要义。

广义而言，“流”（stream）指随时间逐步可用之数据。此概念见于多处：Unix 之`stdin` 与 `stdout`、编程语言中之惰性列表（lazy lists）[^2]、文件系统 API（如 Java 的 `FileInputStream`）、TCP 连接、互联网音视频传输等。

本章将聚焦*事件流*（event stream）作为一种数据管理机制：即上章所论批数据之无界、渐进式对应物。首先，析流之表征、存储与网络传输；继而在[“数据库与流”](/zh-cn/ch12#sec_stream_databases)一节中，探流与数据库之关系；终在[“流之处理”](/zh-cn/ch12#sec_stream_processing)一节中，研持续处理流之方法与工具，及其构建应用之路径。
## 事件流传输 {#sec_stream_transmit}
在批处理世界中，作业的输入与输出皆为文件（或存于分布式文件系统）。流式处理之对应物，何如？

输入若为文件（即字节序列），首道处理工序，常为解析其为记录序列。于流式处理语境中，此“记录”多称 *事件*（event），实则同义：一小而自足、不可变之对象，载有某时刻所发生之事之细节。事件通常含一时间戳，标示其发生之时（依挂钟时间，参见【单调时钟与挂钟时间】(/zh-cn/ch9#sec_distributed_monotonic_timeofday)）。

例如，所发生之事，或为用户行为——如浏览网页、完成购买；或为机器产出——如温度传感器之周期性读数、CPU 利用率指标。在【以 Unix 工具做批处理】(/zh-cn/ch11#sec_batch_unix) 一例中，Web 服务器日志之每一行，即为一事件。

事件可编码为文本字符串、JSON，抑或某种二进制格式（参见第 5 章 /en/ch5#ch_encoding）。此编码允其持久化：譬如追加至文件、插入关系表、写入文档数据库；亦允其跨网传输，送达另一节点以供处理。

批处理中，文件仅写一次，而后可被多个作业读取。类比而言，流式术语中，事件由一 *生产者*（producer，亦称 *发布者*（publisher）或 *发送方*（sender））生成一次，继而可被多个 *消费者*（consumer，亦称 *订阅者*（subscriber）或 *接收方*（recipient））[^3] 处理。文件系统中，文件名标识一组相关记录；流式系统中，相关事件通常聚为一 *主题*（topic）或 *流*（stream）。

原则上，文件或数据库已足以连接生产者与消费者：生产者将其所生每条事件写入数据存储；各消费者周期轮询该存储，检视自上次运行以来新增之事件。此即批处理作业每日终了处理当日数据之本质。

然若转向低延迟之持续处理，而数据存储未专为此类轮询而设，则轮询成本剧增。轮询愈频，返回新事件之请求占比愈低，开销遂愈高。故更优之道，在于事件新增时主动通知消费者。

传统数据库对此类通知机制支持甚弱：关系型数据库虽有 *触发器*（trigger），可响应变更（如表中插入一行），然其能力极为受限，且于数据库设计中本属次要考量 [^4]。故专用工具应运而生，专司事件通知之递送。
### 消息系统 {#sec_stream_messaging}
通知消费者新事件的常见方式，是采用**消息系统**（messaging system）：生产者将事件封装为消息发出，系统将其推送给消费者。此前我们在【事件驱动架构】(/zh-cn/ch5#sec_encoding_dataflow_msg) 中已简要提及此类系统；本节将深入探讨。

以 Unix 管道或 TCP 连接为代表的**直连通信信道**，虽可简单实现消息传递，但多数消息系统均在此基础模型上加以扩展。关键区别在于：Unix 管道与 TCP 仅支持一对一通信（单发单收），而消息系统允许多个生产者向同一主题（topic）发送消息，并允许多个消费者从该主题接收消息。

此即**发布/订阅模型**（publish/subscribe model）。不同系统对此模型的实现路径各异，无普适最优解。为厘清差异，宜聚焦以下两个核心问题：

1. **若生产者发送速率超过消费者处理能力，系统如何应对？**  
   大致有三策：丢弃消息、在队列中缓冲消息、施加**反压机制**（backpressure，亦称流控），即阻塞生产者继续发送。  
   例如，Unix 管道与 TCP 即采用反压：其内置固定大小的小缓冲区；一旦填满，发送方即被阻塞，直至接收方从中取走数据（参见【网络拥塞与排队】(/zh-cn/ch9#sec_distributed_congestion)）。  

   若选择队列缓冲，则须明察队列持续增长时的行为：  
   - 队列溢出内存时，系统是否崩溃？抑或自动落盘？  
   - 若落盘，磁盘 I/O 将如何影响消息系统的性能 [^5]？  
   - 若磁盘写满，系统又将如何响应 [^6]？

2. **若节点宕机或临时离线，消息是否丢失？**  
   如同数据库，持久性（durability）常需结合落盘与/或复制（replication）实现（参见侧栏【复制与持久性】(/zh-cn/ch8#sidebar_transactions_durability)），然此举必有开销。  
   若应用可容忍偶发消息丢失，则通常可在同等硬件上换取更高吞吐与更低延迟。

消息丢失是否可接受，高度依赖具体场景：  
- 对周期性上报的传感器读数或监控指标而言，偶有一两点缺失未必致命——因更新值不久即至；  
- 然须警惕：若大量消息被丢弃，指标失真可能难以即时察觉 [^7]；  
- 若用于事件计数，则可靠性至关重要：每条丢失消息，皆致计数错误。

第十一章所述批处理系统（[Chapter 11](/zh-cn/ch11#ch_batch)）具一优良特性：任务失败后自动重试，且失败任务产生的部分输出自动丢弃。故最终结果与全程无故障时完全一致，从而简化编程模型。本章后文将探讨：如何在流式处理场景中提供类似可靠性保证。
#### 生产者直连消费者之私信机制 {#id296}
若干消息系统采用生产者与消费者之间的直接网络通信，不经过中间节点：

- UDP 组播在金融行业广泛应用，例如股票行情推送等对低延迟要求严苛的场景。尽管 UDP 协议本身不可靠，但应用层协议可恢复丢失的数据包（生产者须缓存已发送的数据包，以便按需重传）。

- ZeroMQ 与 nanomsg 等无代理（brokerless）消息库亦采此路径，基于 TCP 或 IP 组播实现发布/订阅模型。

- 某些指标采集代理（如 StatsD）使用不可靠的 UDP 消息，从全网机器收集并监控指标。（StatsD 协议中，计数器类指标仅当全部消息抵达时方为准确；采用 UDP 仅能保证指标至多近似。参见【TCP 与 UDP 之辨】(/zh-cn/ch9#sidebar_distributed_tcp_udp)。）

- 若消费者对外暴露网络服务，生产者可直接发起 HTTP 或 RPC 请求（参见【服务间数据流：REST 与 RPC】(/zh-cn/ch5#sec_encoding_dataflow_rpc)），将消息推送给消费者。此即 Webhook 之本意：一服务将其回调地址注册于另一服务，后者于事件发生时主动请求该地址。

此类直连消息系统虽在其适用场景下表现良好，然其应用代码须显式处理消息丢失之可能。其容错能力甚为有限：即便协议可检测并重传网络中丢失的数据包，仍普遍假设生产者与消费者始终在线。

若消费者离线，则其不可达期间所发消息必致丢失。部分协议允许生产者重试失败投递，然若生产者自身崩溃，其待重试消息缓冲区亦随之丢失，此机制即告失效。
#### 消息代理 {#id433}
一种广泛应用的替代方案，是通过*消息代理*（亦称*消息队列*，message broker）发送消息。该组件本质上是一种专为处理消息流而优化的数据库[^12]，以服务端形式运行，生产者与消费者均作为客户端与其连接。  
生产者将消息写入代理；消费者则从代理中读取消息。

通过将数据集中于代理，此类系统可更从容地应对客户端的动态上下线（连接、断开、崩溃），并将持久性（durability）责任转移至代理自身。部分消息代理仅将消息暂存于内存；另一些（依配置而定）则将其落盘，以确保代理崩溃时消息不丢失。面对处理缓慢的消费者，代理通常允许无界队列（unbounded queueing）——即不丢弃消息，亦不施加反压（backpressure）；但此行为仍取决于具体配置。

队列机制亦导致消费者通常呈*异步*特性：生产者发送消息后，通常仅等待代理确认已缓存该消息，而不等待消费者完成处理。消息投递至消费者的时间点不确定——多数情况下在数百毫秒内完成，但若队列积压严重，则可能显著延迟。
#### 消息代理与数据库之较 {#id297}
某些消息代理甚至可通过 XA 或 JTA 参与两阶段提交协议（参见[“跨异构系统的分布式事务”](/zh-cn/ch8#sec_transactions_xa)）。此能力使其在行为上近似数据库；然消息代理与数据库之间，仍存若干关键实践差异：

- 数据库通常持久保存数据，直至显式删除；而部分消息代理于消息成功投递至消费者后即自动清除。此类代理不宜用作长期数据存储。

- 因消息被快速清除，多数消息代理假定其工作集较小——即队列长度有限。若因消费者处理迟缓需缓冲大量消息（内存不足时或溢出至磁盘），则单条消息处理耗时上升，整体吞吐量随之下降 [^5]。

- 数据库常支持二级索引及基于查询语言的多种检索方式；消息代理则多提供按模式匹配订阅主题子集的能力。二者本质均为客户端筛选所需数据之机制，但数据库通常具备更强大的查询功能。

- 数据库查询结果通常基于某一时点的数据快照；若另一客户端随后写入新数据并改变该查询结果，首客户端无法感知先前结果已过期（除非重发查询或轮询变更）。相较之下，消息代理不支持任意查询，亦不允许已发送消息更新；但可于数据变动时（即新消息就绪时）主动通知客户端。

此为消息代理之传统范式，已凝练于 JMS [^13] 与 AMQP [^14] 等标准之中，并由 RabbitMQ、ActiveMQ、HornetQ、Qpid、TIBCO Enterprise Message Service、IBM MQ、Azure Service Bus 及 Google Cloud Pub/Sub [^15] 等软件实现。虽可用数据库模拟队列，然调优以达高性能并非易事 [^16]。
#### 多消费者 {#id298}
当多个消费者从同一主题读取消息时，主要采用两种消息传递模式，如[图 12-1](/zh-cn/ch12#fig_stream_multi_consumer) 所示：

**负载均衡（Load Balancing）**

：每条消息仅投递给 *一个* 消费者，使各消费者分担该主题消息的处理任务。代理可任意将消息分配给消费者。此模式适用于消息处理开销较大之场景，便于通过增加消费者实现并行处理。（在 AMQP 中，可通过多个客户端消费同一队列实现；在 JMS 中称为 *共享订阅*（shared subscription）。）

**扇出（Fan-out）**

：每条消息投递给 *所有* 消费者。扇出模式允许多个独立消费者各自“收听”同一消息广播，互不干扰——其流式语义等价于多个批处理作业同时读取同一输入文件。（JMS 中由主题订阅（topic subscription）提供；AMQP 中由交换器绑定（exchange binding）实现。）
{{< figure src="/fig/ddia_1201.png" id="fig_stream_multi_consumer" caption="Figure 12-1. (a) Load balancing: sharing the work of consuming a topic among consumers; (b) fan-out: delivering each message to multiple consumers." class="w-full my-4" >}}
二者可并用，例如借助 Kafka 的 *消费者组（consumer groups）* 机制。  
当一消费者组订阅某主题（topic）时，该主题中每条消息仅投递至该组内某一消费者（组内负载均衡）。  
若两个独立消费者组同时订阅同一主题，则每条消息分别投递至各组内某一消费者（实现跨组扇出，fan-out）。
#### 致谢与重传 {#sec_stream_reordering}
消费者随时可能崩溃，故 broker 或向消费者投递消息后，该消费者尚未处理，或仅部分处理即崩溃。为防消息丢失，消息中间件采用**确认机制（acknowledgments）**：客户端须显式告知 broker 消息已处理完毕，broker 方可将其从队列中移除。

若客户端连接中断或超时，且 broker 未收到确认，则默认该消息未被处理，遂将之重新投递给另一消费者。（须注意：此场景下，消息或已**完全处理完毕**，仅确认包在网络中丢失。此时若需严格保障“恰好一次”语义，须引入原子提交协议，详见【恰好一次消息处理】(/zh-cn/ch8#sec_transactions_exactly_once)；否则，仅当操作本身幂等，或系统无需恰好一次语义时，方可容忍此偏差。）

负载均衡与重投递机制叠加，将影响消息顺序。如[图12-2](/zh-cn/ch12#fig_stream_redelivery)所示：消费者通常按生产者发送顺序处理消息。然消费者2在处理消息 *m3* 时崩溃，恰与消费者1处理 *m4* 同步发生。未确认之 *m3* 遂被重投至消费者1，致其处理序列为 *m4*、*m3*、*m5*。故 *m3* 与 *m4* 之交付顺序，异于生产者1之发送顺序。
{{< figure src="/fig/ddia_1202.png" id="fig_stream_redelivery" caption="Figure 12-2. Consumer 2 crashes while processing m3, so it is redelivered to consumer 1 at a later time." class="w-full my-4" >}}
即便消息代理（message broker）通常力求保持消息顺序（此为 JMS 与 AMQP 标准所要求），负载均衡与重投递（redelivery）并用，终致消息乱序。欲避此弊，可为每一消费者独设一队列（即弃用负载均衡机制）。若消息彼此完全独立，则乱序无害；然若消息间存因果依赖，则乱序将致严重后果，后文详述。

重投递亦或致资源浪费、资源饥饿，乃至流式处理永久阻塞。典型场景如下：生产者序列化消息有误，例如 JSON 编码对象中遗漏必需字段。任一消费者读取该消息时，皆预期该字段存在；若缺失，则处理失败，亦不发送确认（acknowledgement）。于是代理反复重发，致使其他消费者接连失败，形成无限循环。若代理强保序，则后续消息永不得处理；若允许乱序，虽可推进其余消息，却持续耗费资源于注定无法确认之消息。

死信队列（Dead Letter Queue, DLQ）为此而设。消息不滞留原队列无限重试，而移至专用队列以解消费者之阻——[^17]，[^18]。运维通常监控死信队列：队中但有一条消息，即为异常。新消息入列，运维人员可择其一：永久丢弃、人工修正后重发、或修改消费者代码以正确处理该类消息。DLQ 已成多数队列系统标配；今 Apache Pulsar（日志型消息系统）与 Kafka Streams（流处理系统）等亦已支持——[^19]。
### 基于日志的消息代理 {#sec_stream_log}
在网络中发送数据包或向网络服务发起请求，本为瞬时操作，通常不留永久痕迹。虽可通过抓包与日志实现持久化记录，但此非默认预期。AMQP/JMS 类消息中间件承袭此瞬时通信范式：纵使暂存消息至磁盘，亦于投递至消费者后迅即删除。

数据库与文件系统则取相反路径：凡写入者，皆视为持久化存储，至少存续至应用显式删除为止。

此思维差异深刻影响派生数据之构建方式。如[第十一章](/zh-cn/ch11#ch_batch)所述，批处理之核心特性在于可重复执行、自由试验处理逻辑，而无损输入数据——盖因输入恒为只读。AMQP/JMS 式消息系统则不然：若确认（acknowledgment）触发消息自代理中删除，则消费行为具破坏性；故不可重放同一消费者而复现相同结果。

若向消息系统新增消费者，其通常仅接收注册之后所发消息；此前消息已湮灭，不可恢复。反观文件与数据库：任一新客户端皆可随时接入，并读取任意历史时刻所写入之数据（只要该数据未被应用显式覆写或删除）。

然则，何以不能融二者之长——兼取数据库之持久化存储能力，与消息系统之低延迟通知机制？此即*基于日志的消息中间件*（log-based message broker）之设计初衷，近年已广为流行。
#### 以日志作消息存储 {#id300}
日志者，磁盘上仅追加之记录序列也。

前文已数言日志：于[第四章](/zh-cn/ch4#ch_storage)中，论其为日志结构化存储引擎与预写式日志（WAL）之基；于[第六章](/zh-cn/ch6#ch_replication)中，述其为复制机制之载体；于[第十章](/zh-cn/ch10#ch_consistency)中，析其为共识协议之一形。

同此结构，亦可构消息代理（message broker）：生产者将消息追加至日志尾端，消费者依序读取日志以获消息。若消费者读至日志末，则挂起待通知——新消息追加即触发唤醒。Unix 工具 `tail -f` 监测文件末尾追加，其理正如此。

欲逾单盘吞吐之限，日志可分片（shard），如[第七章](/zh-cn/ch7#ch_sharding)所论。各片可布于异机，自成独立日志，读写互不牵制。主题（topic）即为此类分片之集合，其中所有分片承载同类型消息。此法见于[图 12-3](/zh-cn/ch12#fig_stream_kafka_partitions)。

Kafka 称分片为*分区*（partition）。 Broker 于每一分区内，为每条消息赋一单调递增之序号，曰*偏移量*（offset）；[图 12-3](/zh-cn/ch12#fig_stream_kafka_partitions)中方框内数字即为此 offset。此序号可立，盖因分区唯可追加，故其内消息具全序；然跨分区则无序可言。
{{< figure src="/fig/ddia_1203.png" id="fig_stream_kafka_partitions" caption="Figure 12-3. Producers send messages by appending them to a topic-partition file, and consumers read these files sequentially." class="w-full my-4" >}}
Apache Kafka **（分布式日志系统）** 与 Amazon Kinesis Streams 均为基于日志的消息代理，其工作原理如下。Google Cloud Pub/Sub 在架构上与其相似，但对外暴露的是类 JMS 的 API，而非日志抽象 **（即不显式暴露偏移量、分区重放、日志压缩等日志语义）**。尽管此类消息代理将全部消息持久化至磁盘，仍可通过跨多机分片实现每秒百万级消息吞吐；并通过在 **多个副本节点间复制消息**、**同步或异步提交副本写入** 实现容错。
#### 日志与传统消息传递之较 {#sec_stream_logs_vs_messaging}
基于日志的方法天然支持扇出（fan-out）消息传递，因多个消费者可独立读取日志而互不干扰——读取消息并不将其从日志中删除。为实现消费者组内的负载均衡，代理（broker）无需将单条消息分发至各客户端，而可将整块日志分片（shard）分配给组内节点。

各客户端继而消费其所分配分片中的**全部**消息。通常，当某消费者被分配一个日志分片后，即以简单单线程方式顺序读取该分片内消息。此粗粒度负载均衡方案存有如下弊端：

- 可参与某主题（topic）消费任务的节点数上限，即为该主题所含日志分片数；因同一分片内消息必投递至同一节点。（虽可设计一种负载均衡机制：令两个消费者均完整读取同一分片，但仅分别处理偶数偏移量与奇数偏移量的消息；或改用线程池并行处理分片消息。然此类做法将显著增加消费者位点（offset）管理复杂度。故一般而言，单线程处理分片更优；若需提升并行度，宜增配分片数量。）

- 若某条消息处理耗时较长，则阻塞该分片后续消息之处理（即“队首阻塞”，head-of-line blocking；参见【性能描述】(/zh-cn/ch2#sec_introduction_percentiles)）。

因此，若消息处理开销较大、需逐条消息级并行化，且消息顺序性要求不高，则 JMS/AMQP 风格的消息代理更为适宜。反之，若消息吞吐量极高、单条处理极快，且消息顺序至关重要，则日志式架构极为适用——[^23]，[^24]。然须注意：两类架构之界限正日益模糊；例如 Kafka 等日志型消息系统现已支持 JMS/AMQP 风格的消费者组，允许多消费者自同一分区（partition）接收消息——[^25]，[^26]。

由于分片日志仅在单一分片内保障消息顺序，故所有需严格保序的消息，必须路由至同一分片。例如，某应用要求特定用户相关事件按固定次序呈现。此目标可通过事件的用户 ID 决定其所属分片实现——即以用户 ID 作为**分区键**（partition key）。
#### 消费者位移量 {#sec_stream_log_offsets}
顺序消费一个分片（shard），便于判定哪些消息已被处理：所有偏移量（offset）小于消费者当前偏移量的消息，均已处理；所有偏移量大于该值者，尚未被读取。故消息代理（broker）无需为每条消息单独维护确认状态，仅需周期性地持久化消费者的偏移量即可。此法大幅降低元数据开销，并利于批量处理与流水线优化，从而提升基于日志的系统吞吐量。然若消费者发生故障，则恢复时将从**最近一次已持久化的偏移量**起始，而非其实际见过的最新偏移量——由此导致部分消息被重复消费。

此偏移量，实与单主数据库复制中常见的**日志序列号**（Log Sequence Number, LSN）高度相似；后者见于“新建副本节点”一节（参见 [《设置新追随者》](/zh-cn/ch6#sec_replication_new_replica)）。在数据库复制中，LSN 使副本（follower）断连后能重新连接主库（leader），并精确续接复制流，不漏写、不重写。此处原理全同：消息代理充任“主库”，消费者则为“副本”。

若某消费者节点宕机，则其所属消费者组（consumer group）内另一节点将接管其所负责的分片，并从**最后已记录的偏移量**开始继续消费。倘若该消费者此前已处理后续若干消息，却尚未提交其偏移量，则这些消息将在重启后被再次处理。本章后文将详述应对该问题之诸种策略。
#### 磁盘空间使用情况 {#sec_stream_disk_usage}
若仅向日志追加写入，终将耗尽磁盘空间。为回收空间，日志实则划分为若干段（segment），旧段定期删除或移至归档存储。（更精细的磁盘空间释放机制见【日志压缩】一节。）

故若消费者处理迟缓，落后过多，致其消费位点（consumer offset）指向已被删除之段，则部分消息将被跳过。换言之，该日志实质实现一有界缓冲区：满则弃旧，即所谓*循环缓冲区*（circular buffer）或*环形缓冲区*（ring buffer）。唯此缓冲区落于磁盘，故容量可观。

试作粗略估算：当前典型大容量硬盘容量为 20 TB，顺序写吞吐达 250 MB/s。若以极限速率持续写入，约 22 小时即填满磁盘，须始删最老消息。是故，单块磁盘所持日志，必可缓存至少 22 小时之消息；若部署多机多盘，总容量与总写带宽同步提升，缓冲能力亦随之增强。实践中，磁盘 seldom 跑满写带宽，故日志通常可缓存数日乃至数周之消息。

今诸多基于日志之消息中间件，已转向对象存储（object storage）以扩展容量，类同前文【由对象存储支撑之数据库】所述架构。Apache Kafka、Redpanda 等支持分层存储（tiered storage），将旧消息卸载至对象存储，并从中直接服务读请求；WarpStream、Confluent Freight、Bufstream 等则全量数据驻留对象存储。此架构除具成本优势外，亦简化数据集成：对象存储中消息以 Iceberg 表格式组织，可令批处理作业与数据仓库任务直读原数据，免于跨系统复制。
#### 当消费者追不上生产者时 {#id459}
在《消息系统》（["Messaging Systems"](/zh-cn/ch12#sec_stream_messaging)）开篇，曾论及消费者无法跟上生产者发送速率时的三种应对策略：丢弃消息、缓冲（buffering）、或施加反压（backpressure）。依此分类，基于日志（log-based）的设计属**缓冲型**，其缓冲区尺寸极大但固定——上限由可用磁盘空间决定。

若消费者滞后过甚，致其所需消息已早于磁盘所保留之最旧消息，则该消息不可读；此时，代理（broker）实为**按缓冲容量截断旧消息**——即超出保留窗口者，等效丢弃。可观测消费者距日志头部（log head）之滞后量（lag），若滞后显著，可触发告警。因缓冲区庞大，运维人员有充足时间定位并修复慢消费者，使其在消息开始丢失前追平进度。

纵使某消费者终致严重滞后而丢失消息，影响亦**仅限该消费者自身**，不波及其他消费者之服务。此为重大运维优势：可于生产环境中**实验性消费日志**，用于开发、测试或调试，而无须过度担忧干扰线上服务。消费者停机或崩溃后，即停止占用计算与网络资源；唯余其**消费位点（consumer offset）** 持久化存储。

此行为迥异于传统消息代理：后者须谨慎清理已停用消费者的队列；否则，闲置队列将持续积压消息，徒耗内存，挤占活跃消费者之资源。
#### 重放历史消息 {#sec_stream_replay}
此前已指出：在 AMQP 与 JMS 风格的消息代理中，消息的处理与确认属破坏性操作，因其直接导致代理端消息被删除；而日志型消息代理中，消费消息则类比文件读取——纯只读操作，不改变日志本身。

其唯一副作用（除消费者自身输出外）仅为消费者偏移量（consumer offset）前移。然该偏移量由消费者自主控制，故可灵活调整：例如，可启动一个新消费者实例，加载昨日的偏移量，并将输出写入另一位置，从而重处理过去一日之全部消息；此过程可任意重复，且每次均可变更处理逻辑。

由此，日志型消息机制更近似上一章所述批处理范式：输入数据与派生数据经由可复现之转换过程明确分离。此特性支持更自由之实验、更便捷之错误与缺陷恢复，故为组织内部数据流集成之良选 [^27]。
## 数据库与流 {#sec_stream_databases}
我们已对消息代理与数据库作若干比较。虽传统上视二者为不同类工具，然基于日志的消息代理，业已成功借鉴数据库思想并应用于消息传递。反向亦可：汲取消息与流式处理之理念，施于数据库。

其一，可将*事件流用作系统记录源*（参见【记录系统与派生数据】(/zh-cn/ch1#sec_introduction_derived)）。此即“事件溯源”（Event Sourcing）——前文【事件溯源与CQRS】(/zh-cn/ch3#sec_datamodels_events) 中已述：不以可变数据模型（通过更新、删除操作修改状态）存储数据，而将每次状态变更建模为不可变事件，写入仅追加日志（append-only log）；所有读优化的物化视图，皆由此类事件派生。若将基于日志的消息代理配置为永不清除旧事件，则其天然适配事件溯源：既采用仅追加存储，又可低延迟通知消费者新事件。

其二，未必须全盘采纳事件溯源；即便采用可变数据模型，事件流对数据库仍具价值。实则，数据库每一次写入本身即为一个事件，可被捕获、持久化与处理。数据库与流之关联，远不止于磁盘上日志文件的物理存储——其本质极为深刻。

例如，复制日志（参见【复制日志的实现】(/zh-cn/ch6#sec_replication_implementation)）即为数据库写入事件之流：由主节点（leader）在执行事务时生成；从节点（follower）依序应用该写入流至本地副本，从而维持数据一致。复制日志中每一事件，即描述一次实际发生的数据变更。

又如，【共用日志之应用】(/zh-cn/ch10#sec_consistency_smr) 所述“状态机复制”（State Machine Replication）原理：若每个事件均表征一次数据库写入，且所有副本以相同顺序处理全部事件，则各副本终将收敛至同一最终状态。（前提：事件处理为确定性操作。）此亦事件流之又一例证。

本节先析异构数据系统中一典型问题，继而探求如何借事件流之思想解之。
### 保持系统同步 {#sec_stream_sync}
如前所述，全书贯穿一理：**无单一系统可通解数据存储、查询与处理之全部需求**。  
实务中，凡稍具规模之应用，必杂用多种技术以适配各异需求：  
- 以 OLTP 数据库应答用户请求；  
- 以缓存加速高频访问；  
- 以全文索引支撑搜索查询；  
- 以数据仓库承载分析任务。  

各系统皆持一份数据副本，其物理表示（representation）各依所务而优化。

然数据既散存于多处，便须维系彼此一致：  
若某条目于数据库中更新，则缓存、搜索索引、数据仓库亦须同步更新。  
数据仓库之同步，常借 ETL 流程（参见【数据仓库】(/zh-cn/ch1#sec_introduction_dwh)），典型做法为：全量导出源库 → 变换清洗 → 批量载入仓库——即批处理模式。  
同理，前文【批处理用例】(/zh-cn/ch11#sec_batch_output) 已示：搜索索引、推荐系统及其他派生数据系统，亦常由批处理构建。

若周期性全库导出过慢，则偶有采用**双写（dual writes）**者：应用代码于数据变更时，显式向各系统分别写入。  
例如：先写数据库，再更搜索索引，继而失效缓存项（或并发执行诸写操作）。

然双写隐患甚重，其一即**竞态条件（race condition）**，图 12-4（/en/ch12#fig_stream_write_order）所示即为其例。  
设两客户端并发更新条目 X：客户端 1 欲置值为 A，客户端 2 欲置值为 B。  
二者均先写数据库，再写搜索索引。  
因时序不利，请求交错如下：  
- 数据库先收客户端 1 写 A，后收客户端 2 写 B → 终值为 B；  
- 搜索索引先收客户端 2 写 B，后收客户端 1 写 A → 终值为 A。  
两系统自此永久不一致，且全程无任何错误发生。
{{< figure src="/fig/ddia_1204.png" id="fig_stream_write_order" caption="Figure 12-4. In the database, X is first set to A and then to B, while at the search index the writes arrive in the opposite order." class="w-full my-4" >}}
除非另设并发检测机制（如前文“检测并发写入”一节所述之 version vector），否则并发写入将完全无法察觉——其一值将悄然覆写另一值。

双写另一弊，在于二者之一或失败、另一却成功。此属容错问题，非并发问题；然其后果亦致两系统状态互异。欲使二者同成或同败，即原子提交（atomic commit）问题，而该问题求解代价高昂（参见“两阶段提交（2PC）”一节）。

若仅有一份单主复制数据库，则主节点决定写入次序，故状态机复制法可在该库各副本间奏效。然图12-4中并无单一主节点：数据库或有其主，搜索索引亦或有其主，二者互不隶属，遂生冲突（参见“多主复制”一节）。

若真能确立唯一主节点——譬如数据库本身——并令搜索索引降为该数据库之从节点，则局面可大为改善。然此在实践中是否可行？
### 变更数据捕获 {#sec_stream_cdc}
多数数据库的复制日志长期被视为其内部实现细节，而非公开 API。客户端本应通过数据库的数据模型与查询语言进行查询，而非解析复制日志、从中提取数据。

数十年来，诸多数据库根本未提供文档化的机制以获取其写入变更日志。正因如此，将数据库中所有变更完整捕获并同步至其他存储技术（如搜索索引、缓存或数据仓库）极为困难。

近来，“变更数据捕获”（Change Data Capture, CDC）渐受重视：即持续观测数据库所写入的全部数据变更，并将其提取为可向其他系统[^28]的形式。若变更能以流式方式实时提供，则尤为有价值。

例如，可捕获数据库变更，并持续将相同变更应用于搜索索引。只要变更日志按原序应用，即可确保搜索索引中的数据与数据库保持一致。搜索索引及任何其他派生数据系统，皆仅为该变更流之消费者。

[图 12-5](/zh-cn/ch12#fig_stream_change_capture) 展示了 CDC 如何解决[图 12-4](/zh-cn/ch12#fig_stream_write_order) 中的并发问题：尽管将 X 设为 A 与设为 B 的两个请求并发抵达数据库，数据库仍须确定其执行顺序，并依此序写入复制日志；搜索索引依同一顺序拾取并应用变更。若需将数据同步至其他系统（如数据仓库），仅需将其作为 CDC 事件流的另一消费者加入即可。
{{< figure src="/fig/ddia_1205.png" id="fig_stream_change_capture" caption="Figure 12-5. Taking data in the order it was written to one database, and applying the changes to other systems in the same order." class="w-full my-4" >}}

#### 实现变更数据捕获 {#id307}
我们可将日志消费者称为**衍生数据系统**（*derived data systems*），如《记录系统与衍生数据》（["Systems of Record and Derived Data"](/zh-cn/ch1#sec_introduction_derived)）一节所述：搜索索引与数据仓库中所存之数据，仅为记录系统（system of record）中数据的另一视图。变更数据捕获（Change Data Capture, CDC）即为确保记录系统中所有变更均同步反映至衍生数据系统的机制，从而保障衍生系统持有数据之准确性。

本质上，CDC 使某一数据库成为**主库**（leader，即变更捕获源），其余数据库则转为**从库**（followers）。基于日志的消息代理（log-based message broker）极适于将变更事件自源数据库传输至衍生系统，因其能严格保序（避免[图 12-2](/zh-cn/ch12#fig_stream_redelivery)所示重排序问题）。

逻辑复制日志（logical replication logs）可用于实现 CDC（参见《逻辑（行级）日志复制》（["Logical (row-based) log replication"](/zh-cn/ch6#sec_replication_logical)）），但面临若干挑战，例如：如何处理模式变更（schema changes）、如何正确建模更新操作（updates）。开源项目 Debezium 即专为应对上述挑战而设。其提供面向 MySQL、PostgreSQL、Oracle、SQL Server、Db2、Cassandra 等多种数据库的**源连接器**（*source connectors*）；这些连接器接入各库之复制日志，将变更以标准事件模式（standard event schema）暴露出来；后续可对消息作转换，并写入下游数据库。Kafka Connect 框架亦提供多种数据库的 CDC 连接器。Maxwell 通过解析 MySQL 的 binlog [^29] 实现类似功能；Oracle 有 GoldenGate；PostgreSQL 则有 pgcapture。

与消息代理类似，CDC 通常为**异步机制**：记录系统数据库在提交变更时，并不等待该变更被消费者应用完毕。此设计之运维优势在于：引入慢速消费者，对记录系统影响甚微；然其弊端亦明：所有因复制延迟（replication lag）引发的问题皆随之而来（参见《复制延迟之问题》（["Problems with Replication Lag"](/zh-cn/ch6#sec_replication_lag)））。
#### 初始快照 {#sec_stream_cdc_snapshot}
若存有数据库所有变更的日志，则可通过重放（replay）该日志，完整重建数据库任意时刻的状态。然实践中，永久保存全部变更日志将耗费过多磁盘空间，且重放耗时过长，故日志须定期截断（truncate）。

以构建全新全文索引为例：其依赖数据库的完整副本；仅应用近期变更日志不足为凭，因未更新之数据项将被遗漏。故若日志不全，须始于一**一致快照**（consistent snapshot），此法前文“设置新从库”一节已述（参见 [`/en/ch6#sec_replication_new_replica`](/zh-cn/ch6#sec_replication_new_replica)）。

该快照须对应变更日志中某一**已知位置（offset）**，以便快照加载完毕后，确知自何处起始应用后续变更。部分 CDC 工具内置快照机制，其余则需手动执行。Debezium 采用 Netflix 提出的 DBLog 水印算法，实现增量式快照 [^30]、[^31]。
#### 日志压缩 {#sec_stream_log_compaction}
若日志历史仅能保留有限时长，则每次新增派生数据系统，皆须执行快照流程。然*日志压缩*（log compaction）为此提供更优替代方案。

前文“日志结构化存储”一节（见 [第4章 §4.2](/zh-cn/ch4#sec_storage_log_structured)）已述及日志压缩，其应用于日志结构化存储引擎（参见 [图4-3](/zh-cn/ch4#fig_storage_sstable_merging) 示例）。其原理至简：存储引擎周期性扫描日志记录，对键（key）相同者，剔除重复项，仅保留各键之最新更新。此举常使日志段（log segment）显著缩小；故压缩过程亦常同步合并日志段，如 [图12-6](/zh-cn/ch12#fig_stream_compaction) 所示。该过程于后台自动运行。
{{< figure src="/fig/ddia_1206.png" id="fig_stream_compaction" caption="Figure 12-6. A log of key-value pairs, where the key is the ID of a cat video (mew, purr, scratch, or yawn), and the value is the number of times it has been played. Log compaction retains only the most value for each key." class="w-full my-4" >}}
在日志结构化存储引擎中，以特殊空值（*墓碑*，tombstone）执行更新，即表示该键已被删除，并于日志压缩（log compaction）时将其移除。然只要某键未被覆写或删除，其记录便永存于日志之中。经压缩后日志所占磁盘空间，仅取决于数据库当前状态，而非历史写入总量。若同一键频繁覆写，则旧值终将被垃圾回收，唯留最新值。

此原理亦适用于基于日志的消息代理与变更数据捕获（CDC）。若 CDC 系统确保每次变更均含主键，且对同一键的所有更新均为覆盖语义（即新值完全取代旧值），则对任一键，仅需保留其最新一次写入即可。

故当需重建派生数据系统（如搜索索引）时，可自日志压缩主题（log-compacted topic）的偏移量 0 处启动新消费者，顺序遍历全部消息。该日志必包含数据库中每一键之最新值（或偶有残留旧值）——换言之，可借此获取数据库全量内容之完整副本，无须另行对 CDC 源数据库执行快照。

Apache Kafka 支持此日志压缩特性。如本章后文所示，该能力使消息代理不仅可用于临时消息传递，亦可承担持久化存储之责。
#### 变更流 API 支持 {#sec_stream_change_api}
当今主流数据库，多将变更流（change streams）作为一等接口原生支持，而非沿袭过往生硬嫁接、逆向工程式的变更数据捕获（CDC）方案。  
关系型数据库如 MySQL 与 PostgreSQL，通常复用其自身副本同步所依赖的复制日志（replication log）来推送变更。  
各大云厂商亦为其托管数据库及数据仓库提供 CDC 能力：例如 Google Cloud 的 Datastream，即为 Cloud SQL 与 BigQuery 等服务提供实时流式数据接入能力。

即便是最终一致性、基于法定人数（quorum）的数据库（如 Cassandra），如今亦已支持 CDC。  
如前文《线性一致性与法定人数》（["Linearizability and quorums"](/zh-cn/ch10#sec_consistency_quorum_linearizable)）所述，客户端写入须持久化至多数节点，方被视为可见。  
然对法定人数写入实现 CDC 颇具挑战——盖因无单一事实源可供订阅；数据是否可见，取决于各读端的一致性偏好。  
Cassandra 规避此困局之法，在于暴露各节点之原始日志段（raw log segments），而非聚合为单一流式变更序列。  
欲消费该数据之系统，须自行读取各节点原始日志段，并依需合并为统一变更流（其逻辑 akin 于法定人数读取器之合并策略）[^32]。

Kafka Connect [^33] 提供通用适配框架，将多种数据库系统的 CDC 工具与 Kafka 无缝集成。  
一旦变更事件流进入 Kafka，即可用于实时更新派生数据系统（如搜索索引），亦可作为后续章节所述流处理系统的输入源。
#### 变更数据捕获与事件溯源之辨 {#sec_stream_event_sourcing}
试较变更数据捕获（CDC）与事件溯源（Event Sourcing）：

二者皆以日志形式持久化状态变更，然抽象层级迥异。

其一，CDC 作用于存储层。  
应用以常规方式写入数据库（可更新、可删除），CDC 则自底层日志（如复制日志）中解析变更，确保提取顺序与实际写入顺序一致，规避图 12-4 所示竞态风险。此法侵入性低：既存系统仅需启用数据库日志功能，应用本身无需感知。

其二，事件溯源作用于领域层。  
应用逻辑显式建基于不可变事件；事件日志为纯追加（append-only），禁用更新或删除。事件须表征业务事实（如“订单已支付”），而非技术细节（如“orders 表中 status 字段由 'pending' 改为 'paid'”）。故采用事件溯源，须重构应用架构，非轻量演进。

优劣取舍，端视场景而定：  
- CDC 易集成，但将数据库 Schema 暴露为隐式契约——列删改即致下游消费者中断，可能引发面向客户的服务中断 [^34]。此问题原囿于数仓 ETL，今因 CDC 常作实时数据流，生产服务亦成消费者，风险陡增。  
- 解法之一为 *Outbox 模式*：另设专用表（Outbox Table），定义独立 Schema，供 CDC 系统读取 [^35]、[^36]；内部域模型 Schema 可自由演进，不受影响。虽属双写，然因两写同处一数据库事务内，可保强一致性，避开了第 12 章“系统同步”节所述分布式事务难题。  
- Outbox 之代价有二：其一，须手动维护内部 Schema 与 Outbox Schema 间映射，易出错；其二，增加数据库写负载，或引性能瓶颈。

至于日志重放与压缩：  
- CDC 日志中，单条更新事件常含记录全量新值；故主键当前状态唯由该键最新事件决定，旧事件可安全裁剪（log compaction）。  
- 事件溯源日志中，事件表征用户意图（高阶语义），非状态快照；后发事件不覆盖先发事件，完整历史不可弃。故无法按主键裁剪，log compaction 不适用。  

故事件溯源系统常辅以状态快照（snapshot）机制：定期保存由事件日志导出的当前状态，以加速读取与崩溃恢复。然此仅为性能优化；设计前提始终是——原始事件应永久保留，且系统须能随时重放全量事件日志以重建状态。此假设之局限，详见“……”节。

### 状态、流与不可变性 {#sec_stream_immutability}
我们在[第11章](/zh-cn/ch11#ch_batch)中已见：批处理得益于其输入文件之不可变性，故可于既有输入文件上安全运行实验性处理任务，而无损原数据。此不可变性原则，亦为事件溯源（Event Sourcing）与变更数据捕获（Change Data Capture, CDC）强大效力之本源。

数据库通常被视作存储应用之当前状态——该表示法专为读取优化，亦最便于响应查询。然状态之本质即变动不居，故数据库必支持插入、更新与删除。

此与不可变性何以相容？

凡变动之状态，皆为历时所生诸事件之结果。例如：当前可售座位列表，乃所处理预订事件之累积；账户当前余额，乃历次存入与支取事件之净效；Web服务器响应时间图，乃所有历史请求之个体响应时间之聚合。

无论状态如何变迁，必有一串因果相继之事件驱动其变。纵使操作屡作屡撤，事件既已发生，即为事实。核心洞见在于：可变状态与仅追加（append-only）之不可变事件日志，并非矛盾，实为同一枚硬币之两面。记录全部变更之日志——即*变更日志*（changelog）——恰为状态随时间演进之完整轨迹。

若习于数学思维，可将应用状态视作事件流对时间之积分（integration），而变更流则为状态对时间之微分（differentiation），如[图12-7](/zh-cn/ch12#fig_stream_integral) [^37]、[^38] 所示。此比方虽有局限（例如：状态之二阶导数似无实际意义），然不失为理解数据本质之有益起点。
{{< figure src="/fig/ddia_1207.png" id="fig_stream_integral" caption="Figure 12-7. The relationship between the current application state and an event stream." class="w-full my-4" >}}
若将变更日志持久化存储，则仅使系统状态具备可重现性。  
若视事件日志为唯一真实数据源（system of record），而将一切可变状态皆视为其派生结果，则系统内数据流转更易推演。  
吉姆·格雷（Jim Gray）与安德烈亚斯·赖特（Andreas Reuter）于1992年指出：

> \[T\]he database is not fundamentally necessary; the log contains all information that exists.  
> The sole justification for maintaining the database (i.e., the current end-of-log state) is to accelerate retrieval operations.

日志压缩（log compaction）乃弥合日志与数据库状态差异之法：仅保留每条记录之最新版本，覆写旧版即行丢弃。
#### 不变事件之优益 {#sec_stream_immutability_pros}
数据库中不可变性（Immutability）古已有之。譬如，会计行业数百年来即奉行不可变原则：交易发生时，仅以追加方式记入*账簿*（ledger）——此乃事件日志，如实记载资金、货物或服务的流转。

账户报表（如损益表、资产负债表）皆由账簿中诸交易累加推导而得 [^40]。

若记账有误，会计人员不删改原记录，而另增一笔冲正交易（如退还误收款项）。错误交易永存于账簿，盖因审计所需。倘据此错误账簿已发布失实报表，则后续会计期间之报表须包含调整项。此法在会计实务中习以为常 [^41]。

虽可审计性于金融系统尤为关键，然于诸多非强监管系统亦大有裨益。若因代码缺陷致数据库写入错误数据，而该数据库又允许覆写旧值，则恢复极难；若数据仅以追加方式写入不可变事件日志，则问题溯源与修复易如反掌。客户服务亦可凭审计日志厘清用户诉求与投诉根由。

不可变事件所载信息，远超当前状态本身。例如电商网站中，用户将某商品加入购物车后又移除：就订单履约而言，后一事件抵消前一事件；然对分析而言，此行为表明用户曾考虑该商品，终弃购——或待日后回购，或已择他品替代。此类信息悉数留存于事件日志；若数据库于商品移出购物车时即物理删除记录，则此信息尽失。
#### 从同一事件日志派生多个视图 {#sec_stream_deriving_views}
此外，将可变状态与不可变事件日志分离，即可从同一事件日志中派生出多种面向读取的表示形式。其原理一如流式数据的多个消费者（见[图12-5](/zh-cn/ch12#fig_stream_change_capture)）：例如，分析型数据库 Druid 即通过此方式直接从 Kafka 摄取数据；Kafka Connect 的 sink 连接器亦可将 Kafka 中的数据导出至各类数据库与索引 [^33]。

显式地将事件日志翻译为数据库，有助于随时间演进应用：若需引入新功能，以新方式呈现既有数据，可基于事件日志构建独立的、读取优化的新视图，与现有系统并行运行，无须修改旧系统。新旧系统共存，通常比在既有系统中执行复杂模式迁移更为稳妥。待所有读取方切换至新系统，且旧系统确无必要后，即可直接关停，回收其资源 [^42]，[^43]。

此种“以写入优化格式写入数据，再按需将其翻译为不同读取优化表示”的思想，即此前已在【事件溯源与 CQRS】(/zh-cn/ch3#sec_datamodels_events) 中介绍过的 *命令查询职责分离*（Command Query Responsibility Segregation, CQRS）模式。该模式未必依赖事件溯源（Event Sourcing）：仅凭变更数据捕获（CDC）事件流，亦可构建多个物化视图 [^44]。

传统数据库与模式设计思路，隐含一谬误：数据之写入格式必与查询格式一致。而一旦可通过翻译机制，将写入优化的事件日志转为读取优化的应用状态，则归一化与反归一化之争（参见【归一化、反归一化与连接】(/zh-cn/ch3#sec_datamodels_normalization)）便几近失效：在读取优化视图中反归一化数据，全然合理；因翻译过程本身即为保障该视图与事件日志一致性的机制。

在【案例研究：社交网络首页时间线】(/zh-cn/ch2#sec_introduction_twitter) 中，我们曾讨论社交网络的首页时间线——即某用户所关注之人近期发布的帖子缓存（类比邮箱）。此亦为读取优化状态之例：首页时间线高度反归一化，因单条帖子需复制至所有关注该用户的读者之时间线中；但扇出服务（fan-out service）藉由同步新帖与新关注关系，使此类冗余可控。
#### 并发控制 {#sec_stream_concurrency}
CQRS 最大弊端，在于事件日志之消费者多为异步；故用户写入日志后，旋即读取衍生视图，或见其写入尚未反映于视图之中。此即前文“读己所写”（["Reading Your Own Writes"](/zh-cn/ch6#sec_replication_ryw)）所论之问题。

一解，乃于追加事件至日志之时，同步更新读取视图。然此举或需跨事件日志与衍生视图之分布式事务，或须等待事件确已生效于视图——二者皆常不可行，故视图更新通常异步。

反观之，由事件日志推导当前状态，反使并发控制趋于简明。多对象事务之需求（参见["Single-Object and Multi-Object Operations"](/zh-cn/ch8#sec_transactions_multi_object)），多源于单次用户操作须同时修改多处数据。而事件溯源下，可将用户操作建模为自包含之事件；该操作仅需一次原子写入——即向日志追加事件——天然可保原子性。

若事件日志与应用状态按相同维度分片（例如：处理客户所属分片 3 之事件，仅需更新应用状态之分片 3），则单线程日志消费者在写入时无需额外并发控制——因其构造上每次仅处理一个事件（亦参见["Actual Serial Execution"](/zh-cn/ch8#sec_transactions_serial)）。日志通过在分片内定义事件之串行顺序，消除了并发之不确定性 [^27]。若某事件涉及多个状态分片，则需稍作扩展，详见第十三章（[Chapter 13](/zh-cn/ch13#ch_philosophy)）。

另须指出：纵未采用事件溯源模型之系统，亦常借不可变性（immutability）实现并发控制。诸多数据库内部以不可变数据结构或多版本数据支持时间点快照（参见["Indexes and snapshot isolation"](/zh-cn/ch8#sec_transactions_snapshot_indexes)）；Git、Mercurial、Fossil 等版本控制系统，亦赖不可变数据维系文件版本历史。
#### 不可变性之局限 {#sec_stream_immutability_limitations}
永久保存所有变更的不可变历史，在多大程度上可行？答案取决于数据集的变更频率（churn）。  
其一，若工作负载以追加为主，极少更新或删除，则实现不可变性甚易。  
其二，若工作负载在相对较小的数据集上频繁更新或删除，则不可变历史将急剧膨胀，碎片化问题凸显，压实（compaction）与垃圾回收（garbage collection）的性能遂成运维稳健性的关键瓶颈 [^45]、[^46]。

除性能考量外，尚有行政或法律原因要求强制删除数据，即便违背不可变原则。例如，欧盟《通用数据保护条例》（GDPR）明定：用户可随时要求删除其个人数据，亦可要求更正错误信息；又如敏感信息意外泄露时，须迅速遏制扩散。

此时，仅向日志追加一条“逻辑删除”事件，声明此前数据作废，实不足够——须重写历史，使该数据形同从未写入。例如，Datomic 称此机制为 *excision* [^47]；Fossil 版本控制系统则称类似功能为 *shunning* [^48]。

真正删除数据，反出人意料地困难 [^49]。盖因副本散存于多处：存储引擎、文件系统乃至 SSD 常采用异地写入（write-in-place 之反），而非覆写原址 [^41]；备份系统更常刻意设为不可变，以防误删或损坏。

一种支持不可变数据删除的方案是 *crypto-shredding*（密码粉碎）[^50]：未来可能删除的数据，预先加密存储；待删除时，仅销毁密钥。加密后数据虽仍存，然无人可解。此法实为转移问题——数据本身仍不可变，而密钥存储却转为可变。

此外，须预先决定密钥粒度：哪些数据共用同一密钥？何时启用新密钥？此决策至关重要，因 crypto-shredding 仅能整批作废（全删或全留），无法选择性删除某密钥所护之部分数据。若为每条数据单独配密钥，则密钥库规模将与主数据等量齐观，不堪其重。更精巧的方案（如 puncturable encryption，穿孔加密）[^51] 可实现密钥解密能力的细粒度撤销，然尚未广为采用。

综而言之，所谓“删除”，实为“增大数据检索难度”，而非“确保数据绝对不可恢复”。然有时不得不为之，详见【立法与行业自律】(/zh-cn/ch14#sec_future_legislation)。
## 流处理 {#sec_stream_processing}
本章迄今已述 streams 之来源（用户活动事件、传感器数据、数据库写入），及其传输方式（直连消息、消息代理、事件日志）。

所余者，唯 stream 既得之后，可施何等处理——即流式处理（流处理（stream processing））。概言之，有三途：

其一，将事件数据写入数据库、缓存、搜索索引等存储系统，供其他客户端查询。如[图12-5](/zh-cn/ch12#fig_stream_change_capture)所示，此法尤宜于使数据库与系统其余部分的变更保持同步——尤其当该流消费者为该数据库唯一写入方时。此即流式场景下对["批处理用例"](/zh-cn/ch11#sec_batch_output)中输出模式的对应。

其二，将事件推送至终端用户，例如发送邮件告警、移动推送通知，或实时流式推送至仪表盘并可视化呈现。此时，人类为流之最终消费者。

其三，以一个或多个输入流为原料，经处理生成一个或多个输出流。流可穿越若干级此类处理阶段所构成之流水线，终达前述其一或其二之终点。

本章余下篇幅专论其三：以流为输入，产出派生流（derived stream）。实现此类处理之代码，谓之 *operator*（算子）或 *job*（作业）。其与[第11章](/zh-cn/ch11#ch_batch)所述 Unix 进程及 MapReduce 作业高度相似，数据流模式亦同：流处理器以只读方式消费输入流，以仅追加（append-only）方式将其输出写入另一位置。

流处理器中分片（sharding）与并行化之模式，亦与第11章所见 MapReduce 及各类数据流引擎几无二致，故此处不赘。基础映射操作（如记录变换、过滤）之行为亦复相同。

唯关键之异于批处理者，在于流永无终结。此一根本差异引发多重后果：如本章开篇所论，对无限数据集而言，排序失其意义，故归并排序连接（sort-merge join；参见["JOIN 与 GROUP BY"](/zh-cn/ch11#sec_batch_join)）不可用；容错机制亦须重构：批处理任务若运行数分钟即失败，可简单重启重算；而流式作业若已持续运行数年，崩溃后回溯至起点重放全量数据，往往不可行。
### 流处理之用途 {#sec_stream_uses}
流式处理久用于监控场景，即组织需在特定事件发生时即时告警。例如：

- 诈骗检测系统须判别信用卡使用模式是否异常突变，若盗用可能性高，则立即止付。

- 交易系统须实时解析金融市场价格变动，依预设规则自动执行买卖指令。

- 制造系统须持续采集工厂设备状态数据，一旦故障，须秒级定位根因。

- 军事与情报系统须动态追踪潜在敌对行为轨迹，若现攻击征兆，即刻触发警报。

此类应用，皆赖复杂模式匹配与多源事件关联分析。然流式处理之用途，近年已逾监控范畴。本节略述诸类应用之异同。
#### 复杂事件处理 {#id317}
*复杂事件处理*（Complex Event Processing，CEP）是20世纪90年代发展起来的一种技术，用于分析事件流，尤适用于需在事件序列中识别特定模式的应用场景。其原理类比正则表达式——正则表达式可在字符串中匹配字符模式，CEP 则可在事件流中依据规则匹配事件模式。

CEP 系统常采用高级声明式查询语言（如 SQL）或图形化界面，定义待检测的事件模式。此类查询提交至处理引擎后，引擎持续消费输入事件流，并在内部维护状态机以执行匹配。一旦匹配成功，引擎即生成一个*复合事件*（complex event），并附带所匹配事件模式的完整上下文信息。

此类系统中，查询与数据的关系，恰与传统数据库相反：数据库通常持久化存储数据，而将查询视为瞬时请求——查询到来时检索匹配数据，执行完毕即丢弃查询；CEP 引擎则长期驻留查询，事件逐条到达时，引擎实时判断当前已见事件是否构成任一预置查询所定义的模式。

典型 CEP 实现包括 Esper、Apama 与 TIBCO StreamBase；分布式流处理系统（如 Flink、Spark Streaming）亦提供基于 SQL 的声明式流查询能力。
#### 流式分析 {#id318}
流式处理另一重要用途，在于对数据流实施*分析*（analytics）。  
复杂事件处理（CEP）与流式分析之界线模糊，然大体而言：  
- CEP 重在识别特定事件序列；  
- 流式分析则偏重对海量事件作聚合运算与统计度量。  

常见用例有三：  
其一、测度某类事件发生速率（单位时间内的频次）；  
其二、计算某值在指定时段内的滚动均值；  
其三、比对当前统计量与历史时段（如同比上周同一时刻），以侦测趋势，或对异常高低指标触发告警。  

此类统计多基于*固定时间窗口*（window）计算。例如：  
- 求服务最近 5 分钟内每秒查询数之均值；  
- 同期求其响应时间之 99 分位值。  
取数分钟为窗，既可滤除秒级无意义抖动，又足以及时反映流量模式之变化。  
此时间聚合区间，即谓之“窗口”；其详参见【时间推理】一节（/en/ch12#sec_stream_time）。

流式分析系统常采用概率算法以节内存：  
- Bloom 过滤器（见【Bloom 过滤器】/en/ch4#sec_storage_bloom_filter），用于集合成员判定；  
- HyperLogLog [^55]，用于基数估算；  
- 多种分位数估计算法（参见【分位数计算】/en/ch2#sidebar_percentiles）。  
概率算法输出近似结果，但内存开销远低于精确算法。  
然须明辨：此近似性仅为可选优化，并非流式处理之固有属性——流处理本身既可精确，亦可近似；选用概率算法，纯属权衡资源与精度之工程取舍 [^56]。

诸多开源分布式流处理框架，设计之初即以分析为要务：  
Apache Storm、Spark Streaming、Flink、Samza、Apache Beam、Kafka Streams [^57]。  
托管服务则有 Google Cloud Dataflow 与 Azure Stream Analytics。
#### 维护物化视图 {#sec_stream_mat_view}
我们已知：数据库变更流可用于同步派生数据系统（如缓存、搜索索引、数据仓库）与源数据库。此类实践即为**物化视图维护**：基于原始数据集构建一种可高效查询的替代视图，并于底层数据变动时持续更新该视图——[^37]。

同理，在事件溯源（event sourcing）中，应用状态通过逐条应用事件日志得以维持；此时应用状态本身亦属一种物化视图。但与流式分析场景不同，仅考察有限时间窗口内的事件通常不足：构建该物化视图原则上需回溯*全部历史事件*（除经日志压缩而废弃者外）。换言之，所需时间窗口须延伸至系统创世之初。

理论上，任一流式处理器皆可用于物化视图维护；然“永久保留事件”之需，与诸多面向分析的框架（其默认仅处理有限时长窗口）之设计假设相悖。Kafka Streams 与 Confluent 的 ksqlDB 支持此类用法，依托 Kafka 的日志压缩能力 [^58]。

> [!TIP] 增量式视图维护（IVM）
> 数据库似天然适配物化视图维护：其本即为持久化完整数据集而设，且多数亦原生支持物化视图。我们在【物化视图与数据立方体】(/zh-cn/ch4#sec_storage_materialized_views) 中已见，典型数据仓库分析查询可物化为 OLAP 立方体。
>
> 遗憾的是，数据库常借批处理作业或按需请求（如 PostgreSQL 的 `REFRESH MATERIALIZED VIEW`）刷新物化视图表。视图系周期性重算，而非随源数据实时更新。此法有二重大缺陷，致其不适用于流式处理场景下的视图维护：
>
> 1.  效率低下：每次更新均重处理全量数据，而其中大部往往未变；
> 2.  数据新鲜度差：源数据之变更，须待下一轮计划更新执行后方反映至物化视图。
>
> 若数据易分区、计算天然具增量性，则可编写数据库触发器以高效更新物化视图。例如，若物化视图统计每日总销售额，则每发生一笔新销售，仅需更新对应日期之行。定制方案虽在少数情形可行，然多数 SQL 查询难以或无法高效转为增量计算。
>
> <a id="sec_stream_ivm"></a>
>
> **增量式视图维护（Incremental View Maintenance, IVM）** 为上述问题之普适解。IVM 技术将关系代数（如 SQL）转化为支持增量计算的算子。算法仅重算并更新变动数据 [^38]、[^59]、[^60]，视图计算效率大增；更新频次亦可大幅提升，从而显著增强数据新鲜度。
>
> Materialize [^61]、RisingWave、ClickHouse 与 Feldera 等数据库均采用 IVM 技术，提供高效增量物化视图。此类系统摄入事件流，实时暴露物化视图：近期事件暂存内存，周期性用于更新磁盘上物化视图；读取时则融合内存中最新事件与磁盘物化数据，呈现统一实时视图。因读取多以 SQL 表达、物化视图常存为 OLAP 格式，故此类系统亦支持第十一章 (/zh-cn/ch11#ch_batch) 所述之大规模数据仓库级查询。
#### 流式搜索 {#id320}
除复杂事件处理（CEP）外，尚需支持基于复杂条件的单事件检索，例如全文搜索查询。

譬如，媒体监测服务订阅新闻机构发布的文章与广播流，实时查找提及特定公司、产品或主题的新闻。此过程先预设搜索查询，再持续将新闻流逐条匹配该查询。若干网站亦具类似功能：例如，房产网站用户可设置条件，一旦市场上出现符合其筛选要求的新房源，即刻获通知。Elasticsearch 的 `percolator` 功能 [^62] 即为此类流式搜索提供一种实现方案。

传统搜索引擎先对文档建立索引，再于索引之上执行查询；而流式搜索则逆转此范式：查询被预先存储，文档流依次“流经”各查询，其机制类同 CEP。最简实现为将每份文档与全部查询逐一比对；但若查询数量庞大，性能将显著下降。为优化效率，可对查询本身亦建索引，从而缩小潜在匹配的查询集合 [^63]。
#### 事件驱动架构与 RPC {#sec_stream_actors_drpc}
在《事件驱动架构》（[“Event-Driven Architectures”](/zh-cn/ch5#sec_encoding_dataflow_msg)）一节中，吾辈曾论及消息传递系统——即以之为远程过程调用（RPC）之替代方案，用作服务间通信之机制，如参与者模型（actor model）所用者。然此类系统虽亦基于消息与事件，通常不视之为流处理器，盖因三端有别：

- 参与者框架（actor frameworks）主司并发控制与分布式模块之协同执行；流处理则主司数据管理。

- 参与者间通信多为瞬时、点对点；事件日志则具持久性，且允许多订阅者消费。

- 参与者可任意通信（含循环式请求/响应），而流处理器通常构为无环流水线：每条流必为某一特定作业之输出，并严格派生于一组明确定义之输入流。

然则，RPC 类系统与流处理之间，确有交叠之域。例如，Apache Storm 提供 *分布式 RPC*（distributed RPC）功能：用户查询可分发至一批节点，该批节点既处理事件流，亦响应查询；查询请求与输入流事件交织处理，结果可聚合后返于用户。（参见《多分片数据处理》（[“Multi-shard data processing”](/zh-cn/ch13#sec_future_unbundled_multi_shard)）。）

亦可借参与者框架处理数据流。然多数此类框架于节点崩溃时**不保证消息投递**，故若无额外重试逻辑，则处理过程不具备容错能力。
### 关于时间的推理 {#sec_stream_time}
流式处理器常需处理时间，尤以分析场景为甚，诸如“近五分钟均值”之类时间窗口屡见不鲜。  
“近五分钟”看似语义明确，实则暗藏歧义，远非表面那般清晰。

批处理中，任务迅疾遍历大量历史事件。若需按时间切分，须依赖事件自身所携时间戳（event time），而非运行批处理的机器系统时钟——因处理启动时刻，与事件实际发生时刻毫无关联。

一批处理可在数分钟内读取整年历史事件；此时关注的时间线，是那一年的历史本身，而非数分钟的计算耗时。且依事件时间戳处理，可确保结果确定：同一输入、重复运行，必得相同输出。

反观诸多流处理框架，则依赖处理节点本地系统时钟（即 *processing time*）划定窗口 [^64]。  
此法简明直接，当事件生成至事件处理之延迟可忽略时，尚属合理。  
然一旦存在显著处理滞后——即事件处理时刻明显晚于其真实发生时刻——该法即告失效。
#### 事件时间与处理时间 {#id322}
处理延迟，缘由甚多：队列积压、网络故障、性能瓶颈致消息代理或处理器争用、流消费者重启，或因容错恢复、修复代码缺陷而重放历史事件。

此外，消息延迟亦致顺序不可预测。例如：用户先发一请求（由 Web 服务器 A 处理），再发第二请求（由服务器 B 处理）。A、B 各自发出对应事件，然 B 之事件反先于 A 到达消息代理。于是流处理器先见 B 事件、后见 A 事件，虽其真实发生次序恰为相反。

若需类比，可参《星球大战》电影：第四集（1977）、第五集（1980）、第六集（1983）先行上映；其后第一、二、三集于 1999、2002、2005 年补拍上映；第七、八、九集则于 2015、2017、2019 年续出 [^65]。若依上映顺序观看，则观影时序（即处理时序）与叙事时序（即事件时序）相悖。（集数如事件时间戳，观影日期如处理时间。）人类尚可容忍此等不一致，而流处理算法须显式设计，方能应对此类时间与顺序偏差。

混淆事件时间（event time）与处理时间（processing time），必致数据失真。例如：某流处理器统计请求速率（每秒请求数）。若该处理器重部署，停机一分钟，重启后集中处理积压事件；此时若依处理时间统计速率，则积压回放阶段将呈现虚假的请求尖峰，实则真实请求速率恒定如常（见[图 12-8](/zh-cn/ch12#fig_stream_processing_time)）。

#### 处理滞后者事件 {#id323}
以事件时间（event time）定义窗口时，一桩棘手之事在于：永难确知某窗口之全部事件是否业已收讫；亦无法断言尚有无迟来之事件。

譬如，将事件按每分钟切分窗口，以统计每分钟请求数。今已统计若干事件，其时间戳皆落于当小时第37分钟内；而系统时钟已前行，新到事件多属第38、39分钟。此时，何以判定第37分钟窗口业已闭合，并输出其计数值？

可设超时机制：若某窗口久未收到新事件，即宣告其就绪。然仍存可能——部分事件滞留于他机缓冲区，或因网络中断而延迟抵达。此类迟至之事件，谓之“拖尾事件”（*straggler*），须容其被后续处理。概言之，可行路径有二 [^1]：

1.  拒收拖尾事件，盖其在常态下占比甚微。可将丢弃事件数记为监控指标，若丢弃量突增，则告警。

2.  发布“修正值”（*correction*）：以含拖尾事件之更新结果重置该窗口值；必要时，须先撤回先前输出。

某些场景下，可用特殊消息声明：“自此时刻起，不再有时间戳早于 *t* 之消息。”消费者据此触发窗口 [^66]。然若多个生产者分驻异机，各自维护其最小时间戳阈值，则消费者须逐一追踪各生产者状态；此时增删生产者，尤为繁难。
#### 汝所用者，何人之钟？{#id438}
为事件分配时间戳，若事件在系统中多处被缓冲，则难度尤甚。  
例如，某移动应用向服务器上报用户行为事件以供统计分析：当设备离线时，应用须将事件暂存于本地；待网络恢复（或数小时、乃至数日之后），再批量上传。对下游消费者而言，此类事件形同严重滞后的“尾部数据”。

此时，事件所附时间戳，本应反映用户交互发生的真实时刻——即依据移动设备本地时钟所录之时间。  
然用户可控设备之钟表常不可信：或因疏忽、或因故意，其时间可能严重偏差（参见[“时钟同步与精度”](/zh-cn/ch9#sec_distributed_clock_accuracy)）。  
相较之下，服务器接收事件之时刻（依服务器时钟所记）虽更准确（因服务器受控于己），却难以刻画用户真实行为。

为校正设备时钟偏差，可记录三类时间戳：

- 事件发生时刻（设备本地时钟）  
- 事件发送时刻（设备本地时钟）  
- 事件到达服务器时刻（服务器时钟）

以第三项减第二项，可估算设备时钟与服务器时钟之偏移量（前提：网络传输延迟远小于所需时间精度）。  
继而将该偏移量施于第一项，即可推估事件真实发生时刻（前提：设备时钟偏移在事件发生至发送期间未变）。

此非流处理独有之困：批处理亦面临完全相同的时间推理难题。  
唯流式场景下，时间流逝更为显见，故此问题尤为凸显。
#### 窗口类型 {#id324}
既明事件时间戳之确定法，次当定时间窗口之划分方式。窗口既立，即可用于聚合运算，如计数、求均值等。常用窗口类型有四：**滚动窗口（Tumbling window）**、**跳跃窗口（Hopping window）**、**滑动窗口（Sliding window）** 与 **会话窗口（Session window）**。

滚动窗口  
:   长度固定，事件必属且仅属一窗。例如，设 1 分钟滚动窗口，则时间戳在 `10:03:00` 至 `10:03:59` 间之事件归入一窗，`10:04:00` 至 `10:04:59` 归入下一窗，依此类推。实现时，可将事件时间戳向下取整至最近分钟，即得所属窗口。

跳跃窗口  
:   长度亦固定，但允窗重叠，以达平滑效果。例如，5 分钟窗口配 1 分钟步长（hop），则首窗覆盖 `10:03:00` 至 `10:07:59`，次窗覆盖 `10:04:00` 至 `10:08:59`，余类推。实现时，可先按 1 分钟滚动窗口分组，再于相邻若干滚动窗上作聚合。

滑动窗口  
:   包含所有彼此时间间隔不超过指定长度之事件。例如，5 分钟滑动窗口可将 `10:03:39` 与 `10:08:12` 两事件纳入同窗（因相隔仅 4 分 33 秒），而滚动或跳跃之 5 分钟窗则不能（因其依赖固定边界）。实现时，须维护一按时间排序之事件缓冲区，并于事件超时后及时剔除。

会话窗口  
:   异于前三者，无预设时长；其以用户为粒度，将时间邻近之事件聚为一会话，当用户静默逾阈值（如 30 分钟无事件），则会话终结。此为网站分析之常见需求。

窗口运算常需暂存状态。其状态规模或恒定，或随数据增长：  
- 计数类操作，无论窗口大小或事件多少，仅需单个计数器，状态恒定；  
- 滑动窗口及流连接（stream join，见下节）则须缓存事件直至窗口闭合，故窗口过大或吞吐过高时，状态量剧增。  
是故，部署流处理任务时，须确保存算节点内存或磁盘容量足以承载该临时状态。
### 流连接 {#sec_stream_joins}
在【JOIN 与 GROUP BY】（/zh/ch11#sec_batch_join）中，已述批处理作业如何依键联结数据集，此类联结亦为数据流水线之关键环节。流式处理将数据流水线泛化为对无界数据集的增量处理，故流上联结（stream joins）需求与批处理完全等同。

然流上事件可随时抵达，此特性使流式联结较批处理更为复杂。为厘清其异同，今析为三类：**流–流联结**（stream-stream join）、**流–表联结**（stream-table join）、**表–表联结**（table-table join）。下文各举一例以明之。
#### 流-流连接（窗口连接）{#id440}
假设网站设有搜索功能，欲探测用户近期搜索的 URL 趋势。每次用户输入查询，即记录一条搜索事件（含查询词与返回结果）；每次用户点击某条搜索结果，即记录一条点击事件（含所点 URL 与会话 ID）。为计算各搜索结果的点击率（CTR），须将搜索事件与点击事件按**同一会话 ID** 关联。广告系统中同类分析亦普遍存在。

用户或中途放弃搜索，致点击永不发生；即便发生，搜索与点击之间的时间间隔亦极不稳定：通常数秒，偶有延至数日乃至数周者（如用户发起搜索后遗忘标签页，日后重开并点击结果）。加之网络延迟不可控，点击事件甚至可能先于搜索事件抵达。故需设定合理关联窗口——例如，仅当搜索与点击时间差 ≤ 1 小时，方视为可关联。

须注意：将搜索详情直接嵌入点击事件，并不等价于事件关联。此法仅覆盖“用户点击了结果”的情形，却遗漏“用户未点击任何结果”的搜索，无法支撑搜索质量评估。准确 CTR 计算，必须同时纳入全部搜索事件与全部点击事件。

实现此类关联，流处理器须维护**状态**：例如，以会话 ID 为索引，缓存最近一小时内所有搜索事件与所有点击事件。每当新事件到达，即插入对应索引，并立即查另一索引是否存在同会话 ID 的匹配事件。若匹配成功，则输出一条“某搜索结果被点击”事件；若搜索事件在窗口内过期而未见匹配点击，则输出一条“某搜索结果未被点击”事件。
#### 流表连接（流数据增强）{#sec_stream_table_joins}
在【连接与分组】（[第11章第2节](/zh-cn/ch11#sec_batch_join)；[图11-2](/zh-cn/ch11#fig_batch_join_example)）中，  
曾示一例批处理作业：将用户行为事件集与用户档案数据库相连接。  
行为事件天然具流式特性；故可于流处理器中持续执行同类连接：输入为含用户ID的行为事件流，输出则为经用户档案信息增强之事件流。此过程习称“**事件富化**”（enriching）。

欲行此连接，流处理器须逐条处理行为事件，依其用户ID查库，并将对应档案信息注入事件。  
查库可借远程数据库查询实现；然如【连接与分组】节所析，此类远程调用易致延迟升高，且或使数据库过载 [^58]。

另一法为：将数据库副本载入流处理器本地，规避网络往返。此技术称“**哈希连接**”（hash join）——若数据量适中，副本可驻内存，以哈希表组织；若较大，则建索引于本地磁盘。

批处理与流处理之别在于：批作业以数据库某一时刻快照为输入；而流处理器长期运行，数据库内容随时间演进，故其本地副本须实时同步。  
此问题可借**变更数据捕获**（CDC）解之：流处理器既订阅行为事件流，亦订阅用户档案库之变更日志。档案新建或更新时，本地副本同步刷新。由此，连接实为两流之连接：行为事件流与档案变更流。

流-表连接（stream-table join）与流-流连接（stream-stream join）高度相似；最大差异在于：  
- 对档案变更流（即“表”），连接所用窗口溯及“时间起点”（概念上无限长），且新版本记录覆盖旧版；  
- 对行为事件流（即“流”），连接或无需维护窗口。
#### 表与表连接（物化视图维护）{#id326}
试观前文“案例研究：社交网络主页时间线”一节所举之例。吾人曾言：用户欲览其主页时间线时，若遍历其所关注之全部用户、逐一检索其最新帖文、再行归并，则开销过大。

故宜设时间线缓存——即按用户划分之“收件箱”，帖文一经发出，即写入相应收件箱；读取时间线，仅需一次查表即可。

此缓存之物化与维护，须响应如下四类事件：

- 用户 *u* 发布新帖，则该帖须追加至所有关注 *u* 之用户的 timeline 中；  
- 用户删除某帖，或注销其账户，则该帖（或其全部帖文）须自所有用户 timeline 中移除；  
- 用户 *u*~1~ 开始关注用户 *u*~2~，则 *u*~2~ 近期所发之帖（依策略限定数量或时限），须补入 *u*~1~ 的 timeline；  
- 用户 *u*~1~ 取消关注 *u*~2~，则 *u*~2~ 所发之帖须自 *u*~1~ 的 timeline 中清除。

欲以流式处理器实现此缓存维护，须提供两类事件流：  
其一为帖文事件流（含发布与删除）；  
其二为关注关系事件流（含关注与取关）。  

流处理器须持有一份数据库，其中存有每位用户的关注者集合（i.e., `followers[u]`），以便在新帖抵达时，精准定位需更新之 timeline。

另观之，此流处理过程，实为维护一物化视图（materialized view），对应如下跨表联查（join）之查询逻辑：  
`SELECT p.* FROM posts p JOIN follows f ON p.user_id = f.followee_id WHERE f.follower_id = ?`。
``` sql
SELECT follows.follower_id AS timeline_id,
  array_agg(posts.* ORDER BY posts.timestamp DESC)
FROM posts
JOIN follows ON follows.followee_id = posts.sender_id
GROUP BY follows.follower_id
```
流之联结，即对应此查询中表之联结。时间线实为该查询结果之缓存，底层表每有变更，即同步更新。

> [!NOTE]  
> 若视流为表之导数（参见[图12-7](/zh-cn/ch12#fig_stream_integral)），并视联结为两表之积 *u·v*，则物化联结之变更流恰满足乘积法则：(*u·v*)′ = *u*′*v* + *u v*′。  
> 换言之：任一帖子之变更，均与当前关注者集联结；任一关注者之变更，均与当前帖子集联结。
#### 连接操作的时间依赖性 {#sec_stream_join_time}
此处所述三类连接（流-流连接、流-表连接、表-表连接）共性甚多：皆需流处理器基于某一连接输入维护部分状态（如搜索与点击事件、用户档案、关注者列表），再以另一连接输入的消息查询该状态。

状态更新事件之顺序至关重要（先关注后取关，与先取关后关注，结果迥异）。在 Kafka 等分片式事件日志中，单一分区（shard）内事件顺序得以保证，但跨流或跨分区之间通常无全局顺序保障。

由此引出一关键问题：若不同流中事件发生时间相近，其处理次序如何确定？以流-表连接为例：若用户更新个人档案，则哪些活动事件将与旧档案连接（即在档案更新前处理），又有哪些将与新档案连接（即在档案更新后处理）？换言之：若状态随时间演进，而连接操作依赖某时刻之状态，则该“某时刻”究竟为何时？

此类时间依赖广泛存在。例如商品销售场景中，发票所适用税率取决于销售地（国家或州）、商品类型及销售日期（因税率随时间调整）。故将销售流与税率表连接时，须取销售发生时刻之税率——若重处理历史数据，此税率常异于当前税率。

若跨流事件顺序无法确定，则连接结果非确定性（nondeterministic）[^70]：同一作业以相同输入重复运行，未必产出相同结果；因各次运行中，输入流事件交错顺序可能不同。

在数据仓库领域，此问题称为*缓慢变化维*（Slowly Changing Dimension, SCD），常见解法为对被连接记录之每一版本赋予唯一标识符：例如每次税率变更，即生成新标识符；发票中则显式存入销售发生时刻所对应的税率标识符 [^71]、[^72]。此举可使连接确定化，但代价是无法执行日志压缩（log compaction），因表中所有历史版本均须保留。另一替代方案为反规范化（denormalization）：将适用税率直接嵌入每条销售事件中。
### 容错性 {#sec_stream_fault_tolerance}
本章末节，论流处理器之容错机制。

前见[第十一章](/zh-cn/ch11#ch_batch)，批处理框架容错甚易：任一任务失败，可即于他机重试，其输出径直丢弃。此透明重试所以可行者，盖因输入文件 immutable，各任务写入独立输出文件，且输出仅于任务成功完成时方对外可见。

尤须注意：批处理之容错机制，可保作业最终输出与全程无故障时完全一致——纵有任务失败，亦不可察。观之若每条输入记录皆**恰处理一次**（exactly once）：无遗漏，无重复。虽实际中重启致部分记录被多次处理，然输出所呈效果，一如仅处理一次。此即所谓“**恰好一次语义**”（*exactly-once semantics*）；然“**实效一次**”（*effectively-once*）实更贴切 [^73]。

流处理亦面临同款容错之问，然处置远非批处理那般直截：**不可待任务终结再令其输出可见**——盖因数据流无穷无尽，永无“处理完毕”之时。
#### 微批处理与检查点机制 {#id329}
一种解法，是将数据流切分为若干小块，视每一块为微型批处理任务。此法称作**微批处理（microbatching）**，Spark Streaming 以此实现流式计算 [^74]。  
批大小通常设为约一秒，乃性能权衡之果：批过小，则调度与协调开销增大；批过大，则流处理器输出结果之延迟拉长。

微批处理亦隐式提供一翻滚窗口（tumbling window），其宽度即批大小（依处理时间划分，非事件时间戳）；凡需更大窗口之作业，须显式地将状态从前一批传递至下一批。

另一变体，见于 Apache Flink，谓**周期性生成状态滚动快照（rolling checkpoints）**，并持久化至可靠存储 [^75]、[^76]。若流算子崩溃，可自最近快照恢复，并丢弃自上次快照至崩溃期间所产输出。快照由消息流中之**屏障（barriers）** 触发，其作用类同微批边界，然不强制限定窗口尺寸。

在流处理框架内部，微批处理与快照机制皆可达成与批处理等价的**严格一次（exactly-once）语义**。然一旦输出逸出框架之外（如写入数据库、发消息至外部消息中间件、发送邮件），框架便无法撤销失败微批所引发之外部副作用。此时，任务重启将致外部副作用重复发生；单靠微批或快照，不足以杜绝此问题。
#### 原子提交再探 {#sec_stream_atomic_commit}
为在发生故障时呈现“恰好一次”（exactly-once）处理语义，须确保：**事件处理所产生的全部输出与副作用，当且仅当该处理成功完成时，方予生效**。  
此类副作用包括：  
- 向下游算子或外部消息系统（含电子邮件、推送通知）所发送之消息；  
- 向数据库所写入之数据；  
- 算子内部状态之变更；  
- 对输入消息之确认（含在基于日志的消息中间件中前移消费者位点 offset）。

上述操作须满足原子性：**要么全部成功，要么全部不发生**；绝不容许彼此失步。  
若此原则似曾相识，盖因我们在第 8 章《恰好一次消息处理》（["Exactly-once message processing"](/zh-cn/ch8#sec_transactions_exactly_once)）中，已就分布式事务与两阶段提交（2PC）机制详述其理。

第 10 章（[Chapter 10](/zh-cn/ch10#ch_consistency)）指出：传统分布式事务实现（如 XA）存在固有缺陷。  
然于受限环境之中，可高效实现此类原子提交机制。  
Google Cloud Dataflow [^66]、[^75]、VoltDB [^77]、Apache Kafka [^78] 与 [^79] 均采此法。  
其异于 XA 者有三：  
- 不追求跨异构技术栈之通用事务；  
- 将状态变更与消息收发均纳入流处理框架内部统一管理，使事务边界内敛；  
- 通过单次事务批量处理多个输入消息，摊薄事务协议开销。
#### 幂等性 {#sec_stream_idempotence}
吾辈之目标，在于弃置任一失败任务之局部输出，以确保其可安全重试，而无重复生效之虞。分布式事务固可达成此目标，然另有一途，即仰赖**幂等性**（idempotence），如前文《持久化执行与工作流》（["Durable Execution and Workflows"](/zh-cn/ch5#sec_encoding_dataflow_workflows)）[^80] 所述。

幂等操作者，谓可多次执行而效果等同于仅执行一次者。例如：于键值存储中删除某键，即为幂等操作（再次删除，无新效应）；而对计数器执行递增，则非幂等（重复递增，致值累加两次）。

纵使某操作本不具天然幂等性，常可藉少量额外元数据使其幂等。譬如：自 Kafka 消费消息时，每条消息皆具持久、单调递增之 offset；向外部数据库写入值时，可将触发该次写入之消息 offset 一并存入。由此即可判别更新是否已应用，从而避免重复执行。

Storm Trident 之中状态管理，亦基于类似思路。然倚重幂等性，须满足若干前提：其一，失败任务重启后，必按原序重放相同消息（日志型消息中间件可保此性质）；其二，处理过程须具确定性；其三，不可有他节点并发更新同一值 [^81]、[^82]。

若由一处理节点故障转移至另一节点，则或需施加**围栏机制**（fencing）（参见《分布式锁与租约》（["Distributed Locks and Leases"](/zh-cn/ch9#sec_distributed_lock_fencing)）），以防“被判定为死亡实则存活”之节点造成干扰。虽有诸般约束，幂等操作仍不失为实现**恰好一次语义**（exactly-once semantics）之高效手段，仅引入微小开销。
#### 故障后重建状态 {#sec_stream_state_fault_tolerance}
任何需维护状态的流处理任务——例如各类窗口聚合（计数、均值、直方图）及用于连接操作的表与索引——皆须确保该状态可在故障后恢复。

其一，可将状态存于远程数据存储并实施复制；然逐条消息均需查询远程数据库，性能常受拖累。  
其二，可将状态驻留于流处理器本地，并周期性复制至持久化介质；故障恢复时，新任务读取最新复制状态，即可续处理而无数据丢失。

例：Flink 周期性捕获算子状态快照，写入分布式文件系统（如 [^75]、[^76]）；Kafka Streams 则将状态变更发送至专用 Kafka 主题，依托日志压缩（log compaction）机制实现复制，原理类同变更数据捕获（change data capture, [^83]）；VoltDB 采用冗余执行策略，于多个节点上重复处理每条输入消息（参见【实际串行执行】(/zh-cn/ch8#sec_transactions_serial)）。

其三，状态未必需显式复制：若可自输入流重建，则复制反成冗余。例如，若状态仅为短时窗内聚合结果，则重放对应窗口内的输入事件，速度或已足够；若状态系数据库之本地副本，且由变更数据捕获机制维护，则亦可依日志压缩后的变更流重建全库。

然上述权衡，悉取决于底层基础设施之性能特征：某些场景下，网络延迟反低于磁盘访问延迟，网络带宽亦可媲美磁盘带宽。故无普适最优解；本地状态与远程状态之优劣，亦随存储与网络技术演进而动态迁移。
## 摘要 {#id332}
本章论事件流之义、其用及处理之法。

流式处理，于若干方面近似第十一章所论批处理，唯批处理作用于有界（定长）输入 [^84]，而流式处理则持续作用于无界（永续）数据流。由此观之，消息代理与事件日志，实为流式场景下之“文件系统”。

吾辈曾较两类消息代理：

AMQP/JMS 风格消息代理  

：代理将单条消息分派予消费者；消费者于成功处理后逐条确认；消息一经确认，即自代理中删除。此法适于异步远程过程调用（参见第五章“事件驱动架构”），如任务队列——此时消息处理次序无关紧要，且无需回溯重读已处理旧消息。

基于日志之消息代理  

：代理按分片（shard）粒度分配消息，同一分片内所有消息恒由同一消费者节点处理，并严格保序投递。并行性藉分片实现；消费者以“偏移量（offset）”记录进度，即最后已处理消息之位置；代理持久化存储消息于磁盘，故可随时回跳并重读历史消息。

此日志式架构，类同数据库复制日志（参见第六章）、日志结构化存储引擎（参见第四章），亦属共识机制之一种（参见第十章）。吾辈见其尤适于流处理系统：此类系统消费输入流，生成衍生状态或衍生输出流。

至于流之来源，吾辈析数端：用户行为事件、传感器周期性读数、外部数据源（如金融领域行情数据），天然具流式特征。复见一要义：数据库写操作本身亦可视为流——藉变更数据捕获（CDC）隐式提取，或藉事件溯源（event sourcing）显式建模，皆可得数据库之完整变更日志（changelog）。日志压缩（log compaction）更使该流保有数据库之全量快照。

以流表征数据库，遂启系统集成之强大可能。搜索索引、缓存、分析系统等衍生数据系统，但持续消费变更日志并施以更新，即可实时保新。甚者，可自初始时刻起重放全量变更日志，从零构建任意新视图。

流式状态维护与消息重放能力，亦为各类流处理框架实现流连接（stream join）与容错之基石。吾辈所论流处理诸用，包括：检索事件模式（复杂事件处理，CEP）、计算滑动/滚动窗口聚合（流式分析）、同步衍生数据系统（物化视图）。

继而论流处理器中时间推理之难：须明辨处理时间（processing time）与事件时间（event time）；亦须应对“迟滞事件”（straggler event）——即窗口逻辑已闭合之后方抵达之事件。

吾辈析流处理中三类连接操作：

流–流连接（stream-stream join）  

：两输入流皆为行为事件流；连接算子于指定时间窗口内搜寻相关事件。例如：匹配同一用户在三十分钟内所作之两项操作。若欲于单一事件流内查找关联事件，则两输入实为同一流，是谓“自连接（self-join）”。

流–表连接（stream-table join）  

：一输入为行为事件流，另一输入为数据库变更日志；该日志维持本地数据库副本之最新状态。对每条行为事件，连接算子查表并输出增强后之事件。

表–表连接（table-table join）  

：两输入皆为数据库变更日志。此时，任一表之每次变更，均与另一表之最新状态相连接；其输出，即为该两表连接所成物化视图之变更流。

终论流处理器之容错与“恰好一次”（exactly-once）语义实现。如批处理然，失败任务之部分输出须弃置。然流处理长时运行、持续产出，不可尽弃其果；故须采细粒度恢复机制，其基……
微批处理、检查点机制、事务、幂等写入。
