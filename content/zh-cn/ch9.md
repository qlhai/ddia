> *意外之事，殊为奇妙：未临其境，浑然不觉。*  
> ——A.A. 米尔恩，《小熊维尼的房子》（1928）

如[“可靠性与容错”](/zh-cn/ch2#sec_introduction_reliability)所述，构建可靠系统，即确保系统整体在发生故障（fault）时仍能持续运行。  
然穷举一切可能之故障并一一应对，实非易事。开发者常倾力于“顺路逻辑”（happy path）——毕竟多数时刻，诸事无碍；而轻忽异常，盖因故障引出诸多边界情形，繁复难理。

若欲系统于故障之下犹可信赖，须彻底转易心念：专志于思虑“何事或败”，纵其概率微渺，亦不可忽。纵使某故障仅百万分之一，然于规模宏大的系统中，此等事件日日必现。资深系统运维者皆知：**凡可败者，终将败之**。

尤须留意：分布式系统之开发，根本异于单机软件之编写——其要在于，故障之途，骤然倍增，且新奇诡谲，层出不穷 [^1] [^2]。  
本章将初窥实践中常见之困局，并厘清何者可倚、何者不可恃。

为明所抗之挑战，今当极尽悲观之思，细察分布式系统中种种可能之失序：  
先究网络之不可靠（[“不可靠的网络”](/zh-cn/ch9#sec_distributed_networks)），再析时钟与时间同步之舛误（[“不可靠的时钟”](/zh-cn/ch9#sec_distributed_clocks)）。  
诸般问题所酿之果，令人迷惘失据；故须探求：如何刻画分布式系统之状态？如何推断已发生之事？（[“知识、真值与谬误”](/zh-cn/ch9#sec_distributed_truth)）  
继而在[第十章](/zh-cn/ch10#ch_consistency)中，将示例剖析：面对上述诸般故障，何以实现容错。
## 故障与局部失效 {#sec_distributed_partial_failure}
单机程序，行止可期：或成，或败。  
偶有瑕疵之软件，或令人疑其“今日气运不佳”（重启即愈者，多属此例），然此实乃拙劣编码所致，非系统本性使然。

单机软件本无须飘摇不定：硬件无恙，则同操作必得同果（即所谓*确定性*）。若硬件有失（如内存位翻、接插松动），则常致全系统崩解（如内核恐慌、蓝屏死机、无法启动）。故软硬俱佳之单机，常态唯二：全功，或全废；鲜有半通半塞之态。

此乃计算机设计之刻意取舍：一旦内生故障，宁可彻底宕机，毋令错果潜行——盖错果难察、难溯、难纠。是以计算机掩去底层物理之混沌，示人以理想化之抽象模型：CPU 指令恒如其名；写入内存或磁盘之数据，恒存不腐，不妄变。如[“硬件与软件故障”](/zh-cn/ch2#sec_introduction_hardware_faults)所述，此实为近似之真——现实中，静默数据损坏、CPU 静默误算确有发生；然其频次极低，故可暂予忽略。

然若程序运行于多机网络之上，则情势迥异。分布式系统中，故障频仍，不可再视而不见；吾人别无选择，唯直面物理世界之芜杂实相。而物理世界之失序，其谱系之广，远超想象。试观此轶事 [^3]：

> 依我有限经验，曾亲历单数据中心内长时网络分区、PDU（配电单元）失效、交换机宕机、整机柜误断电、数据中心骨干网中断、整数据中心断电，乃至一名低血糖司机驾福特皮卡撞毁数据中心 HVAC（暖通空调）系统。须知，我并非运维人员。  
> —— Coda Hale

分布式系统中，常有局部失常而余部如常者，是谓*部分故障*（partial failure）。其难处在于：此等故障具*非确定性*。凡涉多节点与网络之操作，或成，或败，成败莫测；甚而，成败本身亦不可知！

此非确定性与部分故障之并存，正是分布式系统艰深之本源 [^4]。  
然若系统能容部分故障，则反开宏阔之境：譬如滚动升级——逐台重启节点以施更新，而全局服务始终在线不辍。故容错非为苟且，实为升维：以不可靠之组件，筑可靠之系统。

然欲容错，先须识错。须穷举各类可能之故障——纵属小概率者，亦不可忽；更当于测试环境中主动诱发此类场景，以验系统之应。分布式系统之道，贵在多疑、持重、戒惧。
## 不可靠网络 {#sec_distributed_networks}
如前所述（见《共享内存、共享磁盘与无共享架构》一节），本书所聚焦之分布式系统，多为**无共享系统（shared-nothing systems）**：即一组经网络互连之独立机器。  
网络为其唯一通信途径；各机器独占其内存与磁盘，彼此不可直访对方内存或磁盘（除非通过网络向远程服务发起请求）。纵使存储层为共享（如 Amazon S3），机器亦须经网络与该共享存储服务交互。

互联网及多数数据中心内部网络（常为以太网）属**异步分组网络（asynchronous packet networks）**。此类网络中，一节点可向另一节点发送消息（即数据包），但网络不保证其抵达时间，亦不保证必能抵达。若发出请求并期待响应，则多种异常情形皆可能发生（部分示例如图 9-1 所示）：

1. 请求已丢失（例如网线被拔除）；  
2. 请求滞留于队列中，稍后方送达（例如网络或接收端过载）；  
3. 远程节点已失效（例如进程崩溃或主机断电）；  
4. 远程节点暂未响应（例如正经历长时间垃圾回收停顿；参见《进程停顿》一节），但稍后将恢复响应；  
5. 远程节点已处理请求，但响应在网络中丢失（例如交换机配置错误）；  
6. 远程节点已处理请求，但响应延迟，稍后方送达（例如网络或本机过载）。
{{< figure src="/fig/ddia_0901.png" id="fig_distributed_network" caption="Figure 9-1. If you send a request and don't get a response, it's not possible to distinguish whether (a) the request was lost, (b) the remote node is down, or (c) the response was lost." class="w-full my-4" >}}
发送方无法判断数据包是否送达：唯一办法是等待接收方回传响应消息，而该响应本身亦可能丢失或延迟。在异步网络中，上述情形无法区分——你所知者，唯“尚未收到响应”而已。若向另一节点发出请求而未获响应，则**根本无法判定其原因**。

通常应对之法为设**超时（timeout）**：逾时即放弃等待，并推定响应不会到达。然超时发生之时，发送方仍无从确知远程节点是否已收悉请求；且若请求尚滞留于某处队列，则**即便发送方已放弃，该请求仍可能最终送达接收方**。
### TCP 之局限 {#sec_distributed_tcp}
网络数据包有最大尺寸限制（通常为数 KB），而许多应用需传输超出单包容量的消息（如请求、响应）。此类应用多采用 TCP（传输控制协议），通过建立**连接**，将大数据流拆分为若干分组，并在接收端重新组装。

> [!NOTE]  
> 本文关于 TCP 的多数论述，亦适用于其较新替代协议 QUIC，以及 WebRTC 中使用的流控制传输协议（SCTP）、BitTorrent 的 uTP 协议等其他传输层协议。与 UDP 的对比参见[“TCP 与 UDP”](/zh-cn/ch9#sidebar_distributed_tcp_udp)。

TCP 常被称作提供“可靠”传输：它可检测并重传丢失分组；识别乱序分组并恢复正确顺序；利用简单校验和检测分组损坏。此外，TCP 还动态调节发送速率——在尽可能加快传输的同时，避免压垮网络或接收端；此机制称为**拥塞控制**（congestion control）、**流量控制**（flow control）或**反压**（backpressure）[^5]。

向套接字（socket）写入数据时，“发送”并非即时发生，而仅是将数据暂存于操作系统管理的发送缓冲区中。当拥塞控制算法判定当前具备发包能力，即从该缓冲区取出一个分组长度的数据，交由网络接口发出。该分组经若干交换机与路由器转发，最终抵达接收端；其操作系统将载荷写入接收缓冲区，并向发送方回送确认分组（ACK）。**仅在此时**，接收端操作系统才通知应用程序：又有新数据到达 [^6]。

故 TCP 虽标榜“可靠”，是否意味着网络不可靠性已无需担忧？否。TCP 仅凭超时未收 ACK 即推定分组丢失，却无法判别是原始分组未达，抑或 ACK 自身丢失。虽可重发，但无法保证重传分组必能抵达；若网线拔出，TCP 亦不能代为插回。最终，在可配置超时后，TCP 将放弃重试，并向应用报告错误。

若 TCP 连接因错关闭（例如远端节点崩溃，或网络中断），则**无法得知远端实际处理了多少数据** [^6]。  
即便 TCP 已确认某分组“送达”，此仅表明远端操作系统内核已收悉，而应用进程可能在处理前即已崩溃。欲确证请求成功，须依赖**应用层自身返回的明确响应** [^7]。

然 TCP 仍极有用：它为跨包大消息的收发提供了便捷抽象。连接建立后，更可复用该连接传送多个请求与响应。常见做法是：先发送一个头部，标明后续消息的字节长度，再紧随其后发送消息本体。HTTP 及诸多 RPC 协议（参见[“服务间数据流：REST 与 RPC”](/zh-cn/ch5#sec_encoding_dataflow_rpc)）皆依此模式运作。
### 实践中的网络故障 {#sec_distributed_network_faults}
数十年来，人类持续构建计算机网络——或可期许，今已通晓如何使其可靠。然事与愿违，迄今未竟全功。既有系统性研究，亦有大量经验性证据表明：网络故障远较预期频发，即便在单一企业所控之数据中心等受控环境中，亦复如是：

* 某中型数据中心之研究表明，月均网络故障约十二起；其中半数致单机失连，另半数致整机架断网。
* 另一研究测得接入交换机（top-of-rack switch）、汇聚交换机（aggregation switch）及负载均衡器（load balancer）等组件之故障率。  
  其发现：冗余组网设备之引入，并不能如预期显著降低故障率，盖因冗余无法防范人为失误（如交换机配置错误）——而此恰为宕机主因。
* 广域光纤链路中断之归因，荒诞而真实：牛群啃咬 [^11]、河狸啃噬 [^12]、鲨鱼撕咬 [^13]；  
  唯鲨鱼所致损伤已渐稀少，缘于海底光缆屏蔽工艺精进 [^14]。  
  人为因素亦屡见不鲜：误操作 [^15]、废料盗取 [^16]、蓄意破坏 [^17]，皆可致断。
* 跨不同云区域之往返时延（RTT），高百分位值可达数分钟之久 [^18]。  
  即便同处一数据中心内，若某交换机软件升级过程中发生异常，触发拓扑重配，则分组延迟亦可逾一分钟 [^19]。  
  故须假定：消息延迟可任意长。
* 通信中断偶呈局部性，其影响因通信对端而异：例如 A 与 B 可通，B 与 C 可通，而 A 与 C 不通 [^20] [^21]。  
  更有奇诡故障：某网卡偶发性丢弃全部入向分组，而出向分组仍可正常发送 [^22]；  
  单向链路可用，绝不保证反向亦然。
* 短暂网络中断，或引发远超原故障持续时间之后续影响 [^8] [^20] [^23]。

> [!TIP] 网络分区（NETWORK PARTITIONS）  
> 若网络因故障致某部分与其余部分隔绝，此即所谓 *网络分区*（network partition）或 *netsplit*；然其本质无异于其他网络中断。  
> 此“分区”与存储系统之分片（sharding）无关——后者亦称 *partitioning*（参见[第七章](/zh-cn/ch7#ch_sharding)）。

纵使汝环境之中网络故障罕见，然其 *可能发生* 一事本身，即要求软件必须具备容错能力。  
凡经网络之通信，皆可失败；此为铁律，无可规避。

若网络故障之错误处理未明确定义、未经充分测试，则后果不堪设想：  
例如集群或陷死锁，网络恢复后仍永久无法响应请求 [^24]；  
甚或误删全部数据 [^25]。  
软件一旦遭遇未预见情形，其行为不可预知。

容错（handling network faults）未必等于容忍（tolerating）：  
若网络常态下高度可靠，则故障期间仅向用户展示错误提示，亦属合理策略。  
然必先明察软件对网络问题之反应，并确保系统可自故障中恢复。  
故宜主动注入网络故障以检验系统行为——此即所谓 *故障注入*（fault injection）；参见[“故障注入”](/zh-cn/ch9#sec_fault_injection)。
### 故障检测 {#id307}
诸多系统需自动识别故障节点。例如：

* 负载均衡器须停止向已宕机节点转发请求（即令其“退出轮转”）；
* 在单主复制的分布式数据库中，若主节点失效，则须将某从节点提升为新主节点（参见[“处理节点宕机”](/zh-cn/ch6#sec_replication_failover)）。

然网络之不确定性，使节点是否正常运转难以确断。仅在特定情形下，方得显式反馈以示异常：

* 若可抵达目标节点所在主机，但目标端口无进程监听（如因进程崩溃），操作系统将主动关闭或拒绝 TCP 连接，并回送 `RST` 或 `ICMP Port Unreachable` 报文；
* 若节点进程已崩溃（或遭管理员终止），而其所在操作系统仍运行，则可通过脚本通知其余节点该故障，从而快速触发故障转移，免于等待超时。例如，HBase 即以此法实现 `graceful shutdown`；
* 若可访问数据中心内网络交换机之管理接口，则可查询链路状态，于硬件层检测故障（如远端主机断电）。然此法不适用于经互联网连接之场景；亦不适用于共享型数据中心（无交换机管理权限）；或当网络故障导致管理接口本身不可达之时；
* 若路由器确知所连 IP 地址不可达，或回送 `ICMP Destination Unreachable` 报文。然路由器并无“魔法般”的故障感知能力——其判断亦受制于网络固有局限，与其余网络参与者无异。

远程节点宕机之快速反馈虽具价值，然不可依赖。一旦异常发生，或于协议栈某层收到错误响应；但通常情形下，应预设：**无任何响应**。此时可重试若干次，静候超时；若逾时未获响应，方可宣告该节点失效。
### 超时与无界延迟 {#sec_distributed_queueing}
若超时是唯一可靠的故障检测手段，则超时值应设为几何？  
惜乎无简明答案。

 timeout 过长，则节点被判定为“死”所需等待过久（其间用户或须空等，或见错误提示）；  
 timeout 过短，则故障检出虽快，却易误判：节点实未宕机，仅因瞬时负载激增（如本机 CPU 尖峰、网络拥塞）而响应迟滞，即被草率宣告死亡。

误判之害甚重：若该节点实为存活，且正执行有副作用之操作（例如发送邮件），而另一节点随即接管，则该操作或被执行两次。此问题详见  
[“知识、真与伪”](/zh-cn/ch9#sec_distributed_truth)、  
[第十章](/zh-cn/ch10#ch_consistency)、  
及[“数据库的端到端原则”](/zh-cn/ch13#sec_future_end_to_end)。

节点一旦被宣告死亡，其职责须移交他者，遂加重余下节点与网络之负载。若系统本已高载，此时误判将雪上加霜：尤当原节点实未宕机，唯因过载而响应缓慢；将其负载强转他人，反致级联失败（极端情形下，各节点互判对方死亡，全系统瘫痪——参见[“过载系统何以不可自愈”](/zh-cn/ch2#sidebar_metastable)）。

试构一理想系统：其网络可保证包延迟上限——任一数据包，或于时间 *d* 内送达，或彻底丢失，绝无逾 *d* 者；  
又设非故障节点必于时间 *r* 内完成任一请求处理。  
则成功请求之响应必不晚于 2*d* + *r*（发包耗时 ≤ *d*，处理耗时 ≤ *r*，回包耗时 ≤ *d*）；  
若逾此限时未得响应，即可确知：网络或远端节点已失效。  
此时，2*d* + *r* 即为可证安全之 timeout 下界。

然现实系统罕具此二保障：异步网络具**无界延迟**（尽力速达，然无最大延迟上界）；  
多数服务实现亦无法承诺请求处理之最坏时间上界（参见[“响应时间保障”](/zh-cn/ch9#sec_distributed_clocks_realtime)）。  
对故障检测而言，“多数时候快”不足为凭：timeout 若设得过低，仅需一次短暂的往返延迟尖峰，即足以令系统失衡。

#### 网络拥塞与队列机制 {#network-congestion-and-queueing}
驾车行于道路，通行时长之变，多因交通拥堵。  
同理，计算机网络中分组延迟之变，亦多源于排队[^27]：

* 若多个节点同时向同一目的地发送分组，网络交换机须将其入队，并依序送入目的链路（如[图9-2](/zh-cn/ch9#fig_distributed_switch_queueing)所示）。在繁忙链路上，分组须等待空闲时隙方得发送（此即所谓*网络拥塞*）。若入流数据过载致交换机队列满溢，则分组被丢弃，须重传——此时网络本身运行正常。  
* 分组抵达目标主机后，若所有 CPU 核心均正忙，则操作系统将该网络请求入队，待应用就绪后再予处理。此等待时长随主机负载而异，可任意延长[^28]。  
* 在虚拟化环境中，运行中的操作系统常因另一虚拟机抢占 CPU 核心而暂停数十毫秒。此间虚拟机无法消费网络数据，故入向数据由虚拟机监视器（VMM）缓存（buffered）[^29]，进一步加剧网络延迟之变异性。  
* 如前所述，为避免网络过载，TCP 限制其发包速率。此机制导致数据在进入网络前，即已在发送端额外排队。
{{< figure src="/fig/ddia_0902.png" id="fig_distributed_switch_queueing" caption="Figure 9-2. If several machines send network traffic to the same destination, its switch queue can fill up. Here, ports 1, 2, and 4 are all trying to send packets to port 3." class="w-full my-4" >}}
此外，TCP 检测到丢包并自动重传时，应用层虽不直接感知丢包事件，却必然感知由此引发的延迟：一为等待超时（timeout）到期，二为等待重传报文被确认（ACK）。
<a id="sidebar_distributed_tcp_udp"></a>
> [!提示] TCP 与 UDP 之辨

部分对延迟敏感的应用（如视频会议、VoIP）选用 UDP 而非 TCP。此为可靠性与延迟波动性之间的权衡：UDP 不实施流量控制，亦不重传丢包，故规避了若干导致网络延迟波动的成因（然仍受交换机队列与调度延迟影响）。

若数据一旦延迟即失效，则 UDP 为优选。例如 VoIP 通话中，丢包往往无暇重传——其承载语音须在毫秒级时限内播放于扬声器；此时重传已无意义，应用当以静音填充该时间槽（致声音短暂中断），并继续处理后续数据流。重试行为实由人层完成：  
（“请重复一遍？刚才声音断了一下。”）

——  

上述诸因素皆加剧网络延迟之波动性。尤当系统负载趋近其容量上限时，排队延迟之分布范围显著扩大：富余容量者可迅疾清空队列；而高负载系统则极易迅速积压长队列。

在公有云及多租户数据中心中，网络链路、交换机、乃至每台机器的网卡与 CPU（虚拟机场景下）均为多客户共享资源。海量数据处理可令网络链路达饱和状态（*saturate*）。因你既无法管控、亦无从观测邻近租户（即“噪声邻居”，*noisy neighbor*）对共享资源之占用，故其资源争用将直接导致网络延迟高度波动。

在此类环境中，超时阈值唯能通过实验确定：需长期、跨多节点测量网络往返时延（RTT）之分布，以刻画延迟波动之统计特性；再结合应用自身容错性与实时性要求，在故障检测延迟与误触发超时之风险间取舍权衡。

更优之策，是摒弃静态配置之固定超时，代之以动态机制：持续采集响应时延及其波动（*jitter*），并依实测时延分布自动调优超时值。Phi Accrual 失败检测器（*Phi Accrual failure detector*）即为此类方法之一，Akka 与 Cassandra 等系统已实际采用 [^32] [^33]。TCP 之重传超时（RTO）机制亦遵循相似原理 [^5]。
### 同步网络与异步网络 {#sec_distributed_sync_networks}
分布式系统本可大为简化，倘若网络能保证数据包在某一固定最大时延内送达，且绝不丢包。然则，何不于硬件层面彻底解决此问题，令网络本身即具可靠性，使软件无须操心？

欲解此问，不妨将数据中心网络与传统固网电话系统（非蜂窝、非VoIP）相较。后者可靠性极高：语音帧延迟、通话中断均极罕见。一次通话，需端到端时延持续稳定，且带宽足以承载语音采样流。若计算机网络亦能达此等可靠性与可预测性，岂不甚善？

拨通电话时，网络即建立一 *电路*：沿双方全程路径，预分配固定且有保障之带宽。该电路自接通起即独占资源，直至通话终结 [^34]。  
例如，ISDN 网络以每秒 4000 帧之恒定速率运行；一旦通话建立，每帧（双向各一）即为其预留 16 比特空间。故通话期间，任一方皆可确保每 250 微秒精确发送 16 比特语音数据 [^35]。

此类网络属 *同步式*：数据纵经多跳路由，亦无排队延迟——因下一跳中专属于该通话的 16 比特空间早已预留。既无排队，则端到端最大时延恒定，是谓 *有界时延*。
#### 网络延迟岂可径求可预测？{#can-we-not-simply-make-network-delays-predictable}
须知，电话网络中的“电路”与 TCP 连接迥然不同：电路乃固定带宽之独占通道，一经建立，他人不得复用；而 TCP 数据包则依网络实时空闲带宽， opportunistically（伺机）传输。TCP 可接纳任意长度之数据块（如一封邮件或一个网页），竭力以最短时延完成传输；其空闲时，几无带宽占用（偶有保活包除外）。

若数据中心网络与互联网采用电路交换，则可在电路建立之初即保证最大往返时延（RTT）。然事实并非如此：以太网与 IP 均属分组交换协议，必经队列缓冲，故网络时延无上界；此二协议本无“电路”之概念。

何以数据中心与互联网皆用分组交换？答曰：二者专为**突发性流量**（bursty traffic）而优化。电路适于音视频通话——此类业务需全程维持近似恒定之比特率；而网页请求、邮件发送或文件传输，并无特定带宽需求，唯求尽快完成。

若强以电路传文件，则须预估带宽。估低，则传输徒然迟缓，致网络资源闲置；估高，则电路无法建立（因网络无法保障所申明之带宽）。是以，以电路承载突发数据，既浪费容量，又拖慢传输。TCP 则不然，其依网络可用容量，动态调节发送速率。

曾有尝试构建兼具电路与分组交换之混合网络。*异步传输模式*（ATM）于 1980 年代曾与以太网竞逐，终仅见用于电信核心交换设备，未获广用。InfiniBand 与此略近：其在链路层实现端到端流控，可大幅削减网络内部队列需求；然遇链路拥塞，仍不免时延 [^37]。

藉由审慎部署 *服务质量*（QoS，含报文优先级与调度）与 *准入控制*（admission control，即对发送端限速），可在分组网上模拟电路行为，或提供**统计意义下之有界时延** [^27] [^34]。新兴网络算法如低时延、低丢包、可扩展吞吐量（L4S），旨在客户端与路由器两级协同缓解队列积压与拥塞控制之弊。Linux 流量控制器（TC）亦允应用程序重排报文优先级，以达 QoS 之效。
<a id="sidebar_distributed_latency_utilization"></a>
> [!提示] 延迟与资源利用率

广而言之，可变延迟实为动态资源划分之结果。

譬如，两台电话交换机之间有一条线路，最多承载 10,000 路并发通话。  
每路接通的电路即占用一个通话时隙；此线路可视作一种资源，最多供 10,000 个用户同时共享。  
该资源以*静态*方式划分：即便此刻仅有一路通话在线，其余 9,999 个时隙全空，该电路仍独占等量带宽，不因负载变化而调整。

相较之下，互联网以*动态*方式共享网络带宽。  
发送方竞相推送数据包，力求尽快通过线路；网络交换设备则逐时刻裁定何者优先转发（即实时分配带宽）。  
此法虽致排队延迟，却使线路利用率趋近极致。线路成本固定，利用率愈高，则每字节传输成本愈低。

CPU 场景亦然：若于单个物理核心上动态调度多个线程，则一者常须在操作系统运行队列中等待，待另一者执行完毕方得继续——故其暂停时长[^38]。  
然此法较静态分配 CPU 周期（见[“响应时间保障”](/zh-cn/ch9#sec_distributed_clocks_realtime)）更善用硬件。  
同理，云平台得以于同一物理机上混跑多客户虚拟机，亦赖此高利用率之利。

若资源静态划分（如专用硬件、独占带宽），则可实现确定性延迟保障；  
然代价是资源利用率下降——换言之，成本更高。  
反之，多租户环境采用动态资源划分，虽提升利用率、降低成本，却引入可变延迟。

故可变延迟非自然律令，实为成本/效益权衡之产物。

--------

然当前多租户数据中心、公有云及互联网通信中，此类服务质量（QoS）机制均未启用。  
现有技术无法对网络延迟或可靠性作出任何承诺：须预设网络拥塞、排队及无界延迟必然发生。  
故超时阈值并无“正确”取值，唯凭实验测定。

互联网服务提供商间之对等互联协议（peering agreements），以及依托边界网关协议（BGP）建立的路由策略，其运作逻辑较接近电路交换，而非 IP 协议本身。  
在此层级，用户可购买专用带宽。  
但互联网路由作用于网络层面，非主机间端到端连接；且其配置周期远长于单次通信会话。
## 不可靠的时钟 {#sec_distributed_clocks}
时钟与时间至关重要。各类应用依赖时钟以不同方式回答如下问题：

1. 此请求是否已超时？  
2. 该服务的响应时间 99 分位值为何？  
3. 过去五分钟内，该服务平均每秒处理多少查询？  
4. 用户在本站停留时长几何？  
5. 本文发布于何时？  
6. 提醒邮件应于何年何月何日何时发送？  
7. 此缓存项何时过期？  
8. 日志文件中该错误消息的时间戳为何？

例 1–4 测量**持续时间**（如：请求发出至响应收到之间的时间间隔）；  
例 5–8 描述**时间点**（即发生于特定日期、特定时刻的事件）。

在分布式系统中，时间处理尤为棘手：通信非瞬时——消息自一机发往另一机，必经网络传输耗时。接收时刻恒晚于发送时刻；然因网络延迟波动不定，其差值不可预知。此特性常致多机协同下事件先后次序难以判定。

此外，网络中每台机器各持一物理时钟，通常为石英晶体振荡器。此类硬件固有误差，故各机对时间之认知略有出入：或快或慢。时钟可部分同步，最常用机制为网络时间协议（NTP），其令本地时钟依一组服务器所报时间校准 [^39]；  
该等服务器自身则从更高精度源（如 GPS 接收器）获取时间。
### 单调时钟与本地时间时钟 {#sec_distributed_monotonic_timeofday}
现代计算机至少配备两类时钟：*系统实时时钟（time-of-day clock）* 与 *单调时钟（monotonic clock）*。二者虽皆度量时间，然用途迥异，须严加区分。
#### 时钟（按日时刻）{#time-of-day-clocks}
系统时钟（time-of-day clock）之用，如其名：依某历法（亦称“墙上时钟时间”，wall-clock time）返回当前日期与时刻。例如，Linux 中之 ``clock_gettime(CLOCK_REALTIME)`` 与 Java 中之 ``System.currentTimeMillis()``，皆返回自“纪元”（epoch）起始至今之秒数（或毫秒数）：即公历（Gregorian calendar）下 UTC 时间 1970 年 1 月 1 日零时零分零秒，不计闰秒。部分系统另择他日为参考起点。

（虽 Linux 称其为“实时钟”（real-time clock），然此名与实时操作系统（real-time OS）无涉，详见[“响应时间保障”](/zh-cn/ch9#sec_distributed_clocks_realtime)。）

系统时钟常借 NTP 同步，故一机所产时间戳（timestamp），理想情形下，与他机所产者等义。然此类时钟亦具若干异常行为，下节详述。尤须注意者：若本地时钟较 NTP 服务器超前过多，则可能遭强制回拨，致时间看似倒退。此类跳变，连同闰秒所致跳变，使系统时钟不宜用于测量经过时间（elapsed time）——**[^40]**。

夏令时（DST）启止亦可引发系统时钟跳变；若恒以 UTC 为时区，则可避此弊，因 UTC 无夏令时之设。

历史上，系统时钟分辨率亦颇粗疏，例如旧版 Windows 系统中，时钟每 10 毫秒跃进一次 —— **[^41]**。近世系统中，此弊已大为缓解。
#### 单调时钟 {#monotonic-clocks}
单调时钟适用于测量持续时间（即时间间隔），例如超时或服务响应时间：Linux 中的 `CLOCK_MONOTONIC` 或 `CLOCK_MONOTONIC_RAW`，以及 Java 中的 `System.nanoTime()`，即为单调时钟之例。其名源于“单调递增”之保证——该时钟恒向前行进（而本地时间时钟可能回跳）。

可于某时刻读取单调时钟值，执行某操作后，再于稍后时刻重读该值；二者之**差值**即为两次读取间所经历之真实耗时——其作用类如秒表，而非挂钟。然其**绝对值**无意义：或为系统启动以来之纳秒数，或为其他任意基准；尤须注意，不同计算机之单调时钟值不可互比，因其基准与定义互不相通。

在含多个 CPU 插槽之服务器上，各 CPU 可能配有独立计时器，彼此未必同步（[^43]）。操作系统通过补偿机制，力求向应用线程呈现统一单调视图，即便线程跨 CPU 调度亦然。然此单调性保障宜审慎看待（[^44]）。

NTP 可依检测结果调整单调时钟之推进速率（此谓“平滑调频”，*slewing*），以校正本地石英振荡器相较 NTP 服务器之快慢偏差。默认允许速率浮动 ±0.05%，但 NTP **不得**使单调时钟发生跃变（前跳或回跳）。单调时钟之分辨率通常甚高：多数系统可达微秒级乃至更优。

于分布式系统中，以单调时钟测度耗时（如超时）通常稳妥，因其不依赖各节点间时钟同步，且对测量微小误差不敏感。
### 时钟同步与精度 {#sec_distributed_clock_accuracy}
单调时钟无需同步；而实时时钟（time-of-day clock）须依 NTP 服务器或其他外部时间源校准，方具实用价值。然今之授时手段，远不如人所期许般可靠精准——硬件时钟与 NTP 均属难驯之物。略举数端如下：

* 计算机所用石英晶振时钟精度甚低：其固有**漂移**（drift），即运行速率偏快或偏慢。漂移量随设备温度变化而异。Google 对其服务器时钟漂移之假设上限为 200 ppm（百万分之二百）[^45]，相当于每 30 秒向服务器重同步一次时，最大漂移达 6 毫秒；若每日仅同步一次，则漂移可达 17 秒。此漂移构成理论精度上限，纵使系统其余部分全无异常，亦不可逾越。

* 若本地时钟与 NTP 服务器偏差过大，NTP 客户端或拒予同步，或强制重置本地时钟 [^39]。此时，观测重置前后时间的应用程序，或见时间倒流，或见时间突跳前移。

* 若某节点因误配防火墙而隔绝于 NTP 服务器之外，该故障或长期未被察觉；其间时钟漂移持续累积，终致各节点间时间差异显著。实务中确有此类案例。

* NTP 同步精度受限于网络延迟：拥塞网络中数据包延迟起伏不定，故其精度存有天然上限。一实验表明，互联网上同步所能达到之最小误差约为 35 毫秒 [^46]；然网络延迟偶发尖峰，可致误差达约一秒。依具体配置，大幅延迟或致 NTP 客户端彻底放弃同步。

* 部分 NTP 服务器本身错误或配置失当，所报时间偏差可达数小时 [^47] [^48]。NTP 客户端虽可通过查询多个服务器并剔除离群值以缓解此问题，然将系统正确性系于“互联网上陌生人所告之时”，终究令人忧惧。

* 闰秒使某分钟长度变为 59 秒或 61 秒，凡未预设闰秒之系统，其计时假设即遭破坏 [^49]。闰秒曾致诸多大型系统崩溃 [^40] [^50]，足见时钟假设之疏漏极易潜入系统。处理闰秒之上策，或为令 NTP 服务器“说谎”——将闰秒调整匀摊至整日执行（此谓**抹除法**（smearing））[^51] [^52]；然实际 NTP 服务器行为各异 [^53]。自 2035 年起，闰秒将永久停用，此患幸将消弭。

* 虚拟机中，硬件时钟经虚拟化，为需高精度计时之应用增添新挑战 [^54]。当 CPU 核心由多台虚拟机共享时，任一虚拟机常被暂停数十毫秒，以让渡资源予他者。对应用而言，此暂停体现为时钟突跳前移 [^29]。若虚拟机暂停数秒，则其时钟或已落后真实时间数秒；而 NTP 却仍可能报告时钟“几近完美同步” [^55]。

* 若软件运行于非完全可控之设备（如移动终端或嵌入式设备），则其硬件时钟几无可信。部分用户甚至刻意篡改硬件时钟之日期与时间，以在游戏等场景中作弊 [^56]。故该时钟所指时刻，或远溯于往昔，或遥跃于未来。

若愿倾注足够资源，极高精度之时间同步确可达成。例如，欧盟《金融工具市场指令 II》（MiFID II）要求高频交易基金之所有时钟须与协调世界时（UTC）同步至 100 微秒以内，以助排查“闪崩”等市场异常，并辅助侦测市场操纵行为 [^57]。

此精度赖三者协同：专用硬件（GPS 接收器和/或原子钟）、精密时间协议（Precision Time Protocol, PTP），以及审慎部署与持续监控 [^58] [^59]。然单倚 GPS 风险甚高：GPS 信号易遭干扰，在某些地点（如军事设施附近）干扰频发 [^60]。部分云服务商已为其虚拟机提供高精度时钟同步服务 [^61]。然时钟同步仍须慎之又慎：若 NTP 守护进程配置失当，或防火墙阻断 NTP 流量，则仅凭漂移即可迅速酿成巨大时钟误差。
### 依赖同步时钟 {#sec_distributed_clocks_relying}
时钟之弊，在其表象简明而实藏诸多陷阱：一日未必恰为 86,400 秒；本地时间钟（time-of-day clock）或可倒退；各节点所持之时，彼此或相去甚远。

前文已述，网络或丢包、或任意延时。纵使网络常态下运行良好，软件设计仍须以网络偶发故障为前提，并须从容容错。时钟亦然：虽大体可靠，健壮之软件必预为之备，以应时钟失准。

其一，时钟失准极易隐匿不彰。若某机 CPU 损毁，或网络配置谬误，则系统多致瘫痪，故迅即见察而得修。反之，若其石英晶振失准，或 NTP 客户端配置有误，则多数功能看似如常，唯其时钟渐行渐远，悄然背离真实时间。倘有软件依赖高精度时钟同步，则后果多为静默而细微之数据丢失，而非显见之崩溃 [^62] [^63]。

故凡用需时钟同步之软件者，必严密监控集群内各机之间之钟差（clock offset）。任一节点若钟差逾限，即当判为失效，自集群中剔除。此等监控，旨在钟坏未酿大祸之前，先察其异。
#### 用于事件排序的时间戳 {#sec_distributed_lww}
今论一典型情境：跨多节点事件排序，倚赖时钟虽易，实则险矣。

譬如，若二客户端并发写入分布式数据库，孰先孰后？何者为新？

图9-3（见 `/en/ch9#fig_distributed_timestamps`）示一危险用法：于多主复制数据库中，贸然采用本地时间戳（此例类同图6-8，见 `/en/ch6#fig_replication_causality`）。  
客户端 A 在节点 1 写入 `x = 1`；该写操作复制至节点 3；  
客户端 B 在节点 3 对 `x` 执行自增（得 `x = 2`）；  
最终，两写操作均复制至节点 2。
{{< figure src="/fig/ddia_0903.png" id="fig_distributed_timestamps" caption="Figure 9-3. The write by client B is causally later than the write by client A, but B's write has an earlier timestamp." class="w-full my-4" >}}
在[图9-3](/zh-cn/ch9#fig_distributed_timestamps)中，当一次写入被复制至其他节点时，其时间戳依据**发起该写入的节点本地时钟（time-of-day clock）**生成。本例中时钟同步精度极高：节点1与节点3之间的时钟偏移（skew）不足3毫秒，此精度通常优于实际部署所能达到的水平。

因写入 *x* = 2 基于先前的 *x* = 1 写入而递增，故预期其时间戳应更大。然[图9-3](/zh-cn/ch9#fig_distributed_timestamps)所示并非如此：*x* = 1 的时间戳为42.004秒，而 *x* = 2 的时间戳反为42.003秒。

如[“最后写入者胜出（丢弃并发写入）”](/zh-cn/ch6#sec_replication_lww)节所述，解决不同节点上并发写入值冲突的一种方法是**最后写入者胜出（LWW）**：对同一键，仅保留时间戳最大的写入，其余均予丢弃。在[图9-3](/zh-cn/ch9#fig_distributed_timestamps)示例中，节点2收到这两条事件后，将错误判定 *x* = 1 为更新值，从而丢弃 *x* = 2 的写入——导致增量操作丢失。

此问题可借如下方式避免：**覆写值时，新值的时间戳必须严格大于被覆写值的时间戳，即使此举使时间戳超前于写入者本地时钟**。但此法需额外执行一次读取以获取当前最大时间戳，带来开销。Cassandra 与 ScyllaDB 等系统要求所有副本在单轮通信内完成写入，故直接采用客户端时钟时间戳，并辅以 LWW 策略 [^62]。此法存在严重缺陷：

* 数据库写入可能神秘消失：若某节点时钟滞后，则其无法覆盖由快时钟节点先前写入的值，直至两节点间时钟偏移量被耗尽 [^63] [^65]；此情形可致任意规模数据被静默丢弃，且应用层无任何错误提示。
* LWW 无法区分两种情形：一是严格顺序发生的快速连续写入（如[图9-3](/zh-cn/ch9#fig_distributed_timestamps)中客户端B的增量操作确在客户端A写入之后），二是真正并发写入（双方互不知晓对方操作）。欲防止因果性（causality）破坏，须引入额外因果追踪机制（如版本向量），详见[“检测并发写入”](/zh-cn/ch6#sec_replication_concurrent)。
* 当时钟精度仅为毫秒级时，两节点独立生成相同时间戳的概率显著上升。此时需附加裁决值（tiebreaker）——例如大随机数——以解决冲突；但该法亦可能导致因果性违反 [^62]。

故虽以“最新”值胜出、余者丢弃之法看似简明，然须明了：“最新”之定义系于本地时钟，而该时钟本身未必准确。即便采用高精度 NTP 同步，仍可能出现发送方依自身时钟标记为100 ms的报文，在接收方时钟下抵达时间为99 ms——即报文“先抵达、后发出”，悖理之象。

NTP 同步能否臻于精确，使此类错序永不发生？答案是否定的：NTP 同步误差本身受限于网络往返时延，并叠加石英晶振漂移等其他误差源。欲确保事件顺序正确，须使时钟误差远小于网络延迟，此条件实不可达。

相较之下，所谓**逻辑时钟（logical clocks）** [^66] 更为稳妥：其基于递增计数器而非石英振荡器，专用于事件排序（参见[“检测并发写入”](/zh-cn/ch6#sec_replication_concurrent)）。逻辑时钟不度量真实时刻或流逝秒数，仅刻画事件相对次序（孰先孰后）。与此相对，度量真实流逝时间的时钟（含时钟时间与时钟单调性时钟）统称**物理时钟（physical clocks）**。逻辑时钟及其在 ID 生成器中的应用，详见[“ID 生成器与逻辑时钟”](/zh-cn/ch10#sec_consistency_logical)。
#### 带置信区间的时钟读数 {#clock-readings-with-a-confidence-interval}
汝或可自机器之实时时钟（RTC）读取微秒乃至纳秒级时间戳；然纵得如此细粒度之读数，亦不意味着该值真能达此精度。实则，其极大概率不能——前已述及，廉价石英晶振之漂移，每分钟即便经局域网 NTP 服务器同步一次，亦常达数毫秒。若 NTP 服务器位于公网，则最佳可达精度约为数十毫秒；遇网络拥塞，误差更易骤升至百毫秒以上。

故不可将时钟读数视作一确定时刻，而应视作一置信区间内之时间范围：例如，系统或有 95% 置信度断言“此刻为本分钟起第 10.3 至 10.5 秒之间”，然无法更精确定位 [^67]。  
若仅知时间误差 ±100 ms，则时间戳中微秒位以下诸数字，实无意义。

该不确定性界（uncertainty bound）可依时间源推算：若计算机直连 GPS 接收器或原子钟，则预期误差由设备自身决定；对 GPS 而言，尚受卫星信号质量制约。若时间源自远程服务器，则不确定性为三者之和（首阶近似，且假设可信该服务器）：自上次与该服务器同步后本地石英钟之预期漂移、NTP 服务器自身不确定性、以及往返该服务器之网络时延。

惜乎多数系统未暴露此不确定性：例如调用 `clock_gettime()` 时，返回值不附带该时间戳之预期误差，故无从判别其置信区间宽为五毫秒抑或五年。

亦有例外：Google Spanner 所用 *TrueTime* API [^45] 及 Amazon ClockBound 明确上报本地时钟之置信区间。当查询当前时间，返回二值：`[earliest, latest]`，即*最早可能时间*与*最晚可能时间*。依其不确定性模型，时钟确知真实当前时间必落于此区间之内。该区间宽度取决于本地石英钟距上一次与更高精度时钟源同步之时间长短。
#### 全局快照的同步时钟 {#sec_distributed_spanner}
在《快照隔离与可重复读》一节中，我们讨论了**多版本并发控制**（MVCC），此为数据库之重要机制，尤适于需同时支持两类事务之场景：其一为小而快的读写事务；其二为大而长的只读事务（如备份或分析任务）。MVCC 使只读事务得以观察数据库之**快照**——即某一时刻之全局一致状态，且无需加锁，亦不干扰读写事务。

通常，MVCC 依赖**单调递增之事务 ID**。若某写操作发生于快照之后（即其事务 ID 大于快照所用 ID），则该写对快照事务不可见。单机数据库中，仅需一简单计数器即可生成事务 ID。

然于分布式数据库——尤其跨多机、跨多地数据中心部署者——欲在全体分片上维持全局单调递增之事务 ID，殊为困难，盖因须强协调。该 ID 必须反映**因果序**：若事务 B 读取或覆写事务 A 所写之值，则 B 的 ID 必须大于 A；否则，快照将失其一致性。当事务细碎而频密，于分布式系统中生成此类 ID，即成不可承受之瓶颈。（此类 ID 生成器，将于《ID 生成器与逻辑时钟》一节详述。）

可否以同步之墙上时钟（wall-clock）时间戳代事务 ID？若时钟同步精度足够高，则时间戳天然满足“后发事务具更大时间戳”之序性要求。然其症结，在于**时钟偏差之不确定性**。

Spanner 即以此法实现跨数据中心之快照隔离：[^68] [^69]。  
其依托 TrueTime API 所报告之**时钟置信区间**（confidence interval），并基于如下观察：设两置信区间各为 *A* = [*A<sub>earliest</sub>*, *A<sub>latest</sub>*] 与 *B* = [*B<sub>earliest</sub>*, *B<sub>latest</sub>*]；若二者不交（即 *A<sub>earliest</sub>* < *A<sub>latest</sub>* < *B<sub>earliest</sub>* < *B<sub>latest</sub>**），则 B 必在 A 之后发生，毫无歧义；唯当区间相交，方无法判定 A、B 之先后。

为确保事务时间戳忠实地反映因果序，Spanner 在提交读写事务前，**主动等待一个置信区间长度之时间**。此举保证：任何可能读取该数据之后续事务，其时间戳必落在足够靠后之位置，从而使其置信区间与本事务之区间不交。为使此等待最短，Spanner 必须将时钟不确定性压至最低；为此，Google 在每个数据中心部署 GPS 接收器或原子钟，使时钟同步误差控制在约 **7 毫秒以内** [^45]。

原子钟与 GPS 接收器于 Spanner 并非绝对必需：关键在于获得可量化的置信区间；高精度时钟源之作用，唯在压缩该区间而已。他系统亦渐采类似思路：例如 YugabyteDB 在 AWS 上运行时可利用 ClockBound [^70]；另有多套系统亦不同程度依赖时钟同步机制 [^71] [^72]。
### 进程暂停 {#sec_distributed_clocks_pauses}
再观分布式系统中危险用钟之一例：设有一数据库，每分片（shard）仅容一主节点（leader），唯主节点可接受写请求。然则，某节点何以自知犹为有效主节点（未被其余节点判为失效），从而可安全受理写操作？

其一法，乃主节点向其余节点申领一 *租约*（lease），此物类锁而具超时机制 [^73]。任一时刻，仅一节点可持此租约；故节点一旦获租，即确知自身于租约有效期内为主节点。欲续任主位，该节点须于租约过期前周期性续租。若节点故障，则续租中止，租约到期后，他节点即可接管。

试拟其请求处理循环，形如：
```js
while (true) {
    request = getIncomingRequest();

    // Ensure that the lease always has at least 10 seconds remaining
    if (lease.expiryTimeMillis - System.currentTimeMillis() < 10000) {
        lease = lease.renew();
    }

    if (lease.isValid()) {
        process(request);
    }
}
```
此代码有二弊：

其一，仰赖时钟同步：租约（lease）过期时间由另一台机器设定（例如取当前时间加 30 秒），而本地线程以其本地系统时钟比对。若两机时钟偏差逾数秒，则行为失常，不可预测。

其二，纵使改用本地单调时钟（monotonic clock），仍存隐患：代码隐含假设 ``System.currentTimeMillis()`` 处测时与 ``process(request)`` 处实际处理请求之间耗时极短。通常如此，故设 10 秒缓冲已足。  
然若执行中突发长暂停（如线程于 ``lease.isValid()`` 行附近停滞 15 秒），则请求处理完成前租约早已过期，他节点或已接任 leader；而本线程浑然不觉，直至下一轮循环方知失效——此时不安全操作（如重复提交、状态覆盖）或已发生。

此等长暂停是否合理？实为常见。成因如下：

其一，线程争用共享资源（如锁、队列）致长时间阻塞。迁至多核机器反可能加剧争用，且此类问题难于诊断 `[^74]`。  
其二，运行时垃圾回收（GC）——如 JVM 的“Stop-the-World”机制——曾致数分钟停顿 `[^75]`；今虽改良，但 GC 暂停仍可观（参见[“限制垃圾回收影响”](/zh-cn/ch9#sec_distributed_gc_impact)）。  
其三，虚拟化环境中，虚拟机可被任意挂起（suspend，内存写盘）与恢复（resume），此即“热迁移”（live migration）之基；暂停时长取决于内存脏页写入速率 `[^76]`。  
其四，终端设备（如笔记本、手机）亦可随时挂起/恢复进程（例：合盖休眠）。  
其五，操作系统上下文切换，或虚拟机监控器（hypervisor）切换至他 VM 时，当前线程可在任意指令点被抢占；VM 场景下，其余 VM 所占 CPU 时间称“steal time”；若系统负载高、就绪队列长，则本线程重启延迟显著。  
其六，同步磁盘 I/O 可致线程阻塞 `[^77]`。更险者：I/O 隐蔽发生——如 Java 类加载器惰性载入类文件，时机不可控；I/O 暂停与 GC 暂停更可叠加延宕 `[^78]`；若磁盘实为网络存储（如 AWS EBS），则延时复受网络抖动支配 `[^31]`。  
其七，若启用交换（swapping / paging），一次普通内存访问亦可触发缺页中断，须从磁盘载入页面；高压内存下，或需先换出他页，极端时系统陷于频繁换页（thrashing），几无有效计算；故服务器常禁用 swap，宁杀进程以保稳定。  
其八，Unix 进程可被 ``SIGSTOP`` 信号挂起（如 Shell 中 Ctrl-Z），即刻剥夺 CPU 时间，直至显式恢复。
恢复执行时，由 `SIGCONT` 触发，随即从中断处继续运行。即便你的环境通常不使用 `SIGSTOP`，运维工程师仍可能意外发送该信号。

上述所有情形均可于任意时刻 *抢占* 正在运行的线程，并于后续某一时刻恢复其执行；而线程自身对此毫无察觉。此问题与单机多线程环境下保障线程安全类似：你无法对执行时序作任何假设，因任意上下文切换与并发行为均可能发生。

在单机上编写多线程代码时，我们拥有较成熟的线程安全工具：互斥锁（mutex）、信号量（semaphore）、原子计数器（atomic counter）、无锁数据结构（lock-free data structure）、阻塞队列（blocking queue）等。遗憾的是，这些工具无法直接迁移至分布式系统——因后者无共享内存，仅有经不可靠网络传递的消息。

分布式系统中的任一节点，必须假定其执行可能于任意时刻（甚至函数执行中途）被暂停相当长时间。暂停期间，系统其余部分持续运行，甚至可能因该节点失联而将其宣告为“死亡”。最终，该节点或重启执行，却全然不觉自身曾陷入休眠，直至稍后检查系统时钟方知。
#### 响应时间保障 {#sec_distributed_clocks_realtime}
在诸多编程语言与操作系统中，线程与进程可能无界暂停，前文已述。此类暂停之因，若竭力优化，或可消除。

然有软件运行于严苛环境：若响应逾时，即致严重损害。譬如操控飞机、火箭、机器人、汽车及其他物理实体之计算机系统，须对传感器输入作出迅捷且可预测之响应。此类系统设有一明确**截止时限**（deadline），软件必于其前完成响应；否则，整系统或致失效。此即所谓**硬实时系统**（hard real-time systems）。

> [!NOTE]  
> 在嵌入式系统中，“实时”（real-time）意指：系统经审慎设计与充分验证，确保在**所有情形下**均满足既定之时间约束。此义迥异于网络语境中“实时”之宽泛用法——后者仅指服务器向客户端主动推送数据、或流式处理等场景，**并无硬性响应时限要求**（参见［第12章］(/zh-cn/ch12#ch_stream)）。

例如，若汽车车载传感器判定正发生碰撞，则安全气囊释放绝不可因气囊控制软件中一次不期而至的垃圾回收（GC）暂停而延迟。

提供实时保障，需整个软件栈各层协同支持：  
- 须采用**实时操作系统**（RTOS），以确保进程可在指定时间间隔内获得确定性的 CPU 时间配额；  
- 所有库函数须明示其**最坏执行时间**（worst-case execution time）；  
- 动态内存分配常被严格限制，甚或完全禁用（虽存实时垃圾收集器，但应用仍须确保不致令 GC 负载过重）；  
- 更须投入巨量测试与实测工作，以确证各项时间保证切实达成。

凡此种种，大幅增加开发负担，并严重收窄可选之编程语言、库及工具范围（盖因绝大多数语言与工具**未提供实时保障**）。故实时系统开发成本极高，多用于安全攸关之嵌入式设备。此外，“实时”非即“高性能”——反言之，实时系统吞吐量或更低，因其须将**响应及时性**置于绝对优先地位（另参见［“延迟与资源利用率”］(/zh-cn/ch9#sidebar_distributed_latency_utilization)）。

对多数服务端数据处理系统而言，实时保障既不经济，亦不适用。故此类系统不得不承受非实时环境所固有之暂停与系统时钟漂移。
#### 限制垃圾回收的影响 {#sec_distributed_gc_impact}
垃圾回收曾是进程停顿的主因之一，  
然今算法精进，调优得当者，停顿多不逾数毫秒。

Java 运行时提供多种收集器：并发标记清除（CMS）、垃圾优先（G1）、Z 垃圾回收器（ZGC）、Epsilon、Shenandoah。  
各器适配不同内存特征：如高频对象创建、大堆内存等。  
Go 则仅提供一简明并发标记清除收集器，依运行时自调优。

若须彻底规避 GC 停顿，可选用无 GC 之语言：  
- Swift 以自动引用计数（ARC）判定内存释放时机；  
- Rust 与 Mojo 借类型系统静态追踪对象生命周期，使编译器可精确推知内存驻留时限。

亦可在带 GC 语言中缓解停顿影响：  
其一，视 GC 停顿为节点之短暂计划性宕机，由其余节点承接客户端请求；  
其二，若运行时可预示即将触发 GC，则应用可即刻停止向该节点派发新请求，待其处理完待决请求后，再于空载时执行 GC——此法可对客户端屏蔽停顿，压低响应时间高百分位值。

又一变体：仅用 GC 管理短命对象（回收迅捷），而定期重启进程，以防长命对象积聚至需全量回收之程度。  
每次仅重启单节点，并于重启前将流量导出，类同滚动升级（参见[第 5 章](/zh-cn/ch5#ch_encoding)）。

诸法皆不能根除 GC 停顿，然可显著削弱其对应用之实际影响。
## 知识、真理与谎言 {#sec_distributed_truth}
本章迄今，已析分布式系统异于单机程序之要者：无共享内存，唯赖不可靠网络以传递消息，且延迟不定；系统或遭局部故障、时钟失准、进程暂停等扰。

此诸问题，若未习分布式系统者，必感茫然失据。网络中一节点，**不得确知**他节点之状——唯能依所收（或未收）消息而揣度。欲知远端节点之态（如其存何数据、是否正常运行等），唯赖与其交换消息。若远端节点不响应，则其状态不可判：盖网络之障，与节点之障，实难区分。

此类讨论，几近哲思之域：吾人于系统中，何者可断为真？何者可断为假？若感知与测度之机制本即不可靠，**吾人所持之确信，其界安在**？软件系统，果当恪守物理世界之律乎？如因果之序，岂可违逆？

幸而，吾人不必穷究生命之义。于分布式系统中，但可明述所采之行为假设（即“系统模型”），并据此设计实际系统，使之契合此模型。算法亦可证于特定模型下正确运行。是故，纵使底层模型所予保障甚微，**可靠行为仍可达成**。

然则，纵使于不可靠模型中令软件行为良好，亦非易事。本章余下篇幅，将深究分布式系统中“知识”与“真”之义，以助吾人思辨：可作何假设？欲予何保证？  
[第十章](/zh-cn/ch10#ch_consistency) 将进而剖析若干分布式算法实例，示其如何于特定假设下，提供特定保证。
### 多数决原则 {#sec_distributed_majority}
设想一网络遭遇非对称故障：某节点可正常接收发往自身的全部消息，  
但其发出的任何消息均被丢弃或延迟 [^22]。  
该节点本身运行完好，亦持续收到来自他节点之请求，  
然其余节点无法听闻其响应。  
经超时等待后，其余节点遂宣告其“死亡”，因久未得其音讯。  
事态如噩梦般展开：此半断连节点被拖入坟场，足踢手抓，嘶喊“吾未死！”——  
然无人能闻其声，葬礼队伍仍面无表情、坚定不移地前行。

稍缓之情形下，该半断连节点或察觉自身所发消息未获他节点确认，  
从而推知网络必有故障。  
然其余节点仍误判其为死，而该节点对此束手无策。

第三种情形：设一节点执行暂停达一分钟。  
其间，无请求被处理，无响应被发出。  
其余节点静候、重试、渐失耐心，终宣告其死，并将其抬上灵车。  
须臾，暂停结束，该节点线程重启，若无其事。  
其余节点愕然：此“已故”节点忽自棺中昂首，神完气足，欣然与旁人攀谈。  
初时，该节点甚至不察已过一分钟，亦不知自身已被判死——  
于其观之，距上次与他节点交谈，不过须臾耳。

此三则寓言之训诫在于：节点不可尽信己之判断。  
分布式系统不可独倚单点，盖因节点随时可能失效，致系统僵滞、无法自愈。  
故多数分布式算法依凭 *法定人数（quorum）*，即节点间投票机制（参见[“读写法定人数”](/zh-cn/ch6#sec_replication_quorum_condition)）：  
重大决策须获若干节点之最低票数，以弱化对任一节点之依赖。

此原则亦适用于“节点死亡宣告”。  
若法定人数之节点共同宣告某节点死亡，则该节点即为死亡，  
纵其自觉生龙活虎，亦须遵从法定人数之裁决，自行退位。

最常见者，为绝对多数法定人数——即逾半数节点（虽亦容他种法定人数）。  
多数法定人数可确保：当少数节点故障时，系统仍可运行（三节点容一故障；五节点容二故障）。  
其安全性亦有保障：系统内至多存在唯一多数派，  
故不可能同时出现两组互斥之多数派并行决策。  
法定人数之详用，将于[第十章](/zh-cn/ch10#ch_consistency)述及 *共识算法（consensus algorithms）* 时再议。
### 分布式锁与租约 {#sec_distributed_lock_fencing}
分布式应用中的锁与租约易被误用，此为常见缺陷之源。  
今析一典型误用情形。

前文《进程暂停》（见第9章第9.2节）已述：租约乃一种带超时机制的锁；若原持有者失联（或因崩溃、或因长时间暂停、或因网络断连），该租约即自动过期，可由新节点接管。凡需系统中“有且仅有一个”实体之场景，皆可借租约实现。例如：

- 数据库分片仅容一节点担任主节点，以防脑裂（参见第6章第6.3节《节点故障处理》）；  
- 某资源或对象仅容一事务或客户端更新，以防并发写入致数据损坏；  
- 大规模批处理任务中，某输入文件仅应由一节点处理，以免多节点重复执行而浪费算力。

然须审慎思之：若因进程暂停等故，多个节点同时自认持有有效租约，后果如何？  
第三例中，仅致算力冗余，危害甚微；  
前两例中，则或致数据丢失、或致数据损坏，后果严重。

例如，《图9-4》（见第9章第9.4节）所示，即为租约实现不当引发的数据损坏缺陷。（此非假想案例：HBase 曾真实存在此问题，已于 [^85] [^86] 修复。）  
设某存储服务要求：任一文件同一时刻仅能被一个客户端访问；否则多客户端并发写入，将致文件损坏。  
为达此目的，设计令客户端须先向锁服务申请租约，方得访问该文件。此类锁服务常基于共识算法实现；详论见第10章《一致性》。

#### 隔离僵尸进程与延迟请求 {#sec_distributed_fencing_tokens}
术语“僵尸”（*zombie*）偶用于指代某前租约持有者——其尚未察觉自身租约已失效，却仍以当前租约持有者自居并继续执行操作。因无法彻底排除僵尸节点之存在，故须确保其无法引发“脑裂”（split brain）等破坏性后果。此即所谓“隔离僵尸”（*fencing off* the zombie）。

部分系统尝试通过关停僵尸节点实现隔离：例如切断其网络连接 [^9]、经云服务商管理接口关闭虚拟机、甚至物理断电 [^87]。此法称作“击毙另一节点”（*Shoot The Other Node In The Head*，STONITH）。然其存三弊：  
其一，不防大时延网络分区，如[图9-5](/zh-cn/ch9#fig_distributed_lease_delay)所示；  
其二，或致全体节点相互关停（即“集体自毁”）[^19]；  
其三，待僵尸被侦知并关停之时，数据或已遭篡改，为时已晚。

更健壮之隔离方案，可同时防御僵尸节点与延迟请求，见[图9-6](/zh-cn/ch9#fig_distributed_fencing)。
{{< figure src="/fig/ddia_0906.png" id="fig_distributed_fencing" caption="Figure 9-6. Making access to storage safe by allowing writes only in the order of increasing fencing tokens." class="w-full my-4" >}}
设锁服务每次授予锁或租约时，皆附带一“围栏令牌”（fencing token），此为单调递增之整数（如由锁服务维护并自增）；客户端向存储服务发起写请求时，须携带其当前持有的围栏令牌。

> [!NOTE]  
> 围栏令牌亦有他称：Google Chubby 锁服务中称 *sequencers* [^88]；Kafka 中称 *epoch numbers*；共识算法（见[第 10 章](/zh-cn/ch10#ch_consistency)）中，Paxos 的 *ballot number* 或 Raft 的 *term number*，功能亦与此类似。

如[图 9-6](/zh-cn/ch9#fig_distributed_fencing) 所示：客户端 1 先获租约，令牌为 33；继而长时间停顿，致租约过期。客户端 2 随即获租约，令牌为 34（严格递增），并向存储服务提交写请求，附令牌 34。稍后，客户端 1 复活，携旧令牌 33 发起写请求；然存储服务已记录更高令牌（34）之写操作，故拒收令牌 33 之请求。凡新获租约者，须立即向存储服务执行一次写操作；该写完成之后，所有“僵尸客户端”即被围栏隔离。

若以 ZooKeeper 为锁服务，可用事务 ID `zxid` 或节点版本号 `cversion` 充当围栏令牌 [^85]；  
若用 etcd，则修订号（revision number）联合租约 ID 可达同等目的 [^89]；  
Hazelcast 的 `FencedLock` API 显式生成围栏令牌 [^90]。

此机制要求存储服务具备判别写请求所持令牌是否过时之能力；退而求其次，亦可仅支持“仅当对象自本客户端上次读取后未被其他客户端修改时，写操作方可成功”——即类原子比较并交换（CAS）语义。例如，对象存储服务多提供此类校验：Amazon S3 称 *conditional writes*，Azure Blob Storage 称 *conditional headers*，Google Cloud Storage 称 *request preconditions*。
#### 多副本场景下的围栏机制 {#fencing-with-multiple-replicas}
若客户端仅需向一种支持条件写入的存储服务写入，则锁服务略显冗余——**盖因租约分配本可直接依托该存储服务实现**。  
然一旦引入围栏令牌（fencing token），即可将其用于多个服务或副本，并确保旧租约持有者在所有相关服务上均被围栏隔离。

例如，设该存储服务为无主复制的键值存储（leaderless replicated key-value store），冲突策略为“后写胜出”（last-write-wins）（参见［“无主复制”］(/zh-cn/ch6#sec_replication_leaderless)）。在此类系统中，客户端直接向各副本发送写请求，各副本依据客户端所附时间戳独立判定是否接受该写入。

如［图 9-7］(/zh-cn/ch9#fig_distributed_fencing_leaderless) 所示，可将写入者的围栏令牌置于时间戳的最高有效位（most significant bits）或最高有效数字位（most significant digits）。由此可确保：新租约持有者生成的任意时间戳，必大于旧租约持有者生成的任意时间戳——纵使后者写入发生于前者之后，亦不例外。
{{< figure src="/fig/ddia_0907.png" id="fig_distributed_fencing_leaderless" caption="Figure 9-7. Using fencing tokens to protect writes to a leaderless replicated database." class="w-full my-4" >}}
在[图9-7](/zh-cn/ch9#fig_distributed_fencing_leaderless)中，客户端2持有围栏令牌（fencing token）34，故其所有以`34…`开头的时间戳，均大于客户端1生成的任何以`33…`开头的时间戳。  
客户端2向多数派副本（quorum）写入成功，但未能抵达副本3。  
因此，当已成“僵尸”的客户端1后续尝试写入时，其写操作仍可能在副本3上成功——尽管副本1与副本2已忽略该请求。  
此情形无害：因后续的多数派读取（quorum read）必优先选取时间戳更大者（即客户端2的写入）；且读修复（read repair）或反熵（anti-entropy）机制终将覆盖客户端1所写之旧值。

由上述可知：不可武断假定任一时刻仅有一个节点持有有效租约（lease）。  
幸而，若谨慎设计，围栏令牌（fencing token）足以阻断僵尸节点及延迟请求造成的数据损坏。
### 拜占庭容错问题 {#sec_distributed_byzantine}
fencing 令牌可检测并阻止节点**无意间**出错（例如，因尚未察觉租约已过期而继续操作）。然若节点**蓄意**破坏系统保障，则仅需伪造 fencing 令牌发送消息，即可轻易绕过。

本书假设节点“不可靠但诚实”：或迟滞、或失联（因故障），其状态或陈旧（因 GC 暂停或网络延迟）；然若其响应，则必据所知为真——恪守协议规则。

若节点可能“说谎”（即任意发送错误或篡改响应），则分布式系统问题陡增。例如，同一轮选举中投出多个矛盾选票。此类行为称作 *Byzantine 故障*（拜占庭故障）；于互不信任环境中达成共识之难题，即 *拜占庭将军问题* [^94]。

> [!TIP] 拜占庭将军问题

拜占庭将军问题是所谓 *两军问题* [^95] 的推广：设两支军队分驻两地，仅能遣信使通信；而信使或迟滞、或失踪（类比网络丢包）。此即第 10 章（[第 10 章](/zh-cn/ch10#ch_consistency)）所论 *共识* 问题。

拜占庭版本中，有 *n* 位将军须达成一致，然其中混有叛徒。多数将军忠诚，所发消息属实；叛徒则可任意伪造、歪曲消息以惑众。叛徒身份事先未知。

拜占庭（Byzantium）为古希腊城邦，后为君士坦丁堡，即今土耳其伊斯坦布尔。史无实据表明拜占庭将军较他处更擅权谋。其名实取 *Byzantine* 之引申义——“过度繁复、官僚化、诡谲难测”，此义早于计算机时代即用于政治语境 [^96]。Lamport 欲择一不冒犯读者之国名，曾被告诫：称 *阿尔巴尼亚将军问题* 并非良策 [^97]。

若一系统在部分节点故障失序、违逆协议，或遭恶意攻击者干扰网络时，仍能正确运行，则谓之 *拜占庭容错*（Byzantine fault-tolerant）。

此关切仅适用于特定场景。例如：

* 航空航天环境：辐射可致内存或 CPU 寄存器数据损坏，使节点响应全然不可预测。系统失效代价极高（如飞机坠毁、全员罹难；或火箭撞毁国际空间站），故飞控系统须具拜占庭容错能力 [^98] [^99]。
* 多方参与系统：部分参与者或意图欺诈。此时，节点不可轻信他方消息——因其或含恶意。例如，比特币及其他区块链，即旨在令互不信任之多方，在无中心权威前提下，就交易是否发生达成共识 [^100]。

然本书所论系统，通常可安全假设：无拜占庭故障。数据中心内，所有节点皆属本组织管控（故可信赖）；辐射水平极低，内存损坏非主要风险（虽轨道数据中心已在探讨中 [^101]）。多租户系统中，租户彼此互不信任，但依赖防火墙、虚拟化与访问控制策略隔离，而非拜占庭容错。拜占庭容错协议开销甚巨 [^102]；容错嵌入式系统则倚赖硬件层支持 [^98]。于多数服务端数据系统，部署拜占庭容错方案成本过高，实不可行。

Web 应用须防客户端（如浏览器）之任意恶意行为——因其受终端用户控制。故输入校验、净化及输出转义至关重要（以防 SQL 注入、跨站脚本等）。然此处通常不用拜占庭容错协议，唯令服务器为唯一裁决者：判定何等客户端行为被允、何等被禁。于点对点网络（P2P），既无中心权威，则拜占庭容错更为相关 [^103] [^104]。
软件之缺陷，或可视为拜占庭故障；然若各节点部署同一软件，则拜占庭容错算法亦无济于事。  
盖多数拜占庭容错算法，须逾三分之二节点正常运作，方能保障系统正确性（例如四节点系统，至多容许一节点失效）。  
欲以此法抵御软件缺陷，须备四套彼此独立之实现，并寄望缺陷仅现于其一——此为推测，非稳健之策。

同理，若协议可天然防御漏洞、安全失陷与恶意攻击，固属理想；然实则不可行：  
因多数系统中，攻击者一旦攻陷一节点，极可能尽陷其余节点——盖其所运行者，实为同一软件。  
故传统机制（身份认证、访问控制、加密、防火墙等），仍为御敌之主干。

#### 撒谎之弱式 {#weak-forms-of-lying}
虽设节点大体可信，然添设防“微谎”之机制，仍属必要：如因硬件故障、软件缺陷或配置失当所致之无效消息。此类防护非完备拜占庭容错（Byzantine Fault Tolerance），不足以抵御蓄意攻击者；然其简捷务实，确为提升系统可靠性之可行进阶。例如：

* 网络数据包偶因硬件异常、操作系统、驱动或路由器之缺陷而损坏。TCP 与 UDP 协议内置校验和通常可捕获此类损坏，然偶有漏检 [^105] [^106] [^107]。  
  应用层协议自设校验和，即足以应对此类损坏；TLS 加密连接亦具抗损坏之效。

* 面向公网之应用，须审慎净化用户输入：如校验数值是否处于合理区间、限制字符串长度，以防因过大内存分配引发拒绝服务（DoS）。防火墙后之内网服务，或可放宽输入校验之严度；然协议解析器中施行基础校验，仍属必要 [^105]。

* NTP 客户端可配置多个时间服务器地址。同步时，客户端遍询各服务器，估算其偏差，并验证多数服务器所报时间是否落于同一合理区间。只要多数服务器正常，配置错误、报时严重偏移之服务器即被识别为离群值（outlier），并被排除于同步之外 [^39]。多服务器设计，使 NTP 比单服务器方案更具鲁棒性。
### 系统模型与现实 {#sec_distributed_system_model}
诸多算法专为求解分布式系统问题而设计——例如，我们将在[第 10 章](/zh-cn/ch10#ch_consistency)中考察共识问题的若干解法。欲使此类算法切实可用，须能容错：即承受本章所论之各类分布式系统故障。

算法须以硬件与软件配置无关之方式编写，不可过度依赖具体运行环境之细节。此要求进而引出一项前提：须对系统中可能发生的故障类型予以形式化刻画。为此，我们定义**系统模型**（*system model*）——即一种抽象，用以明确算法可作何假设。

就时序假设而言，常用三类系统模型：

**同步模型**  
该模型假定：网络延迟有界、进程暂停有界、时钟偏差有界。此非谓时钟完全同步或网络延迟为零；仅指网络延迟、进程暂停与时钟漂移均恒不逾某固定上界 [^108]。然此模型不符多数实际系统之实情，盖因（如本章所述）无界延迟与暂停确会发生。

**部分同步模型**  
部分同步意谓：系统*多数时刻*表现如同步系统，但偶有突破网络延迟、进程暂停以及时钟偏差之上界者 [^108]。此乃众多系统之合理建模：日常情形下，网络与进程行为良好——否则万事皆不可为；然须直面一事实：任何时序假设皆可能偶遭颠覆。一旦发生，网络延迟、暂停与时钟误差皆可任意大。

**异步模型**  
此模型禁止算法作任何时序假设；甚者，算法甚至无时钟可用（故不可设超时）。虽有算法可适配异步模型，然其约束极严。

此外，除时序问题外，尚须考量节点失效。常见节点系统模型如下：

**崩溃停止故障**（Crash-stop faults）  
在*崩溃停止*（或*失效停止*）模型中，算法仅需考虑一种节点失效方式：即节点崩溃 [^109]。  
意即：节点可在任意时刻突停响应，此后永不再现——既不恢复，亦不重连。

**崩溃恢复故障**（Crash-recovery faults）  
假设节点可在任意时刻崩溃，并于未知时长后重启响应。在此模型中，节点被假定具备**稳定存储**（即非易失性磁盘存储），其内容跨崩溃得以保留；而内存状态则视为全失。

**性能退化与功能降级**  
除崩溃与重启外，节点尚可“变慢”：或仍能响应健康检查请求，却已迟滞至无法执行实质任务。例如：  
- 千兆网卡因驱动缺陷骤降至 1 Kb/s 吞吐 [^110]；  
- 内存压力下进程多耗时于垃圾回收 [^111]；  
- 磨损固态硬盘性能飘忽不定；  
- 高温、松动接口、机械振动、电源异常、固件缺陷等硬件问题亦可致性能劣化 [^112]。  
此类情形称作**跛行节点**（*limping node*）、**灰度故障**（*gray failure*）或**慢失效**（*fail-slow*） [^113]，其处置难度常甚于彻底失效之节点。相关问题尚有：进程部分功能停滞而其余照常——例如后台线程崩溃或死锁 [^114]。

**拜占庭故障**（Byzantine / arbitrary faults）  
节点行为全无约束，可任意妄为，包括蓄意欺骗、误导他节点，如前节所述。

对真实系统建模而言，**部分同步模型 + 崩溃恢复故障**通常最为实用：它容许网络延迟无界、进程暂停无界、节点响应迟滞。然则，分布式算法如何在此模型下运作？
#### 定义算法的正确性 {#defining-the-correctness-of-an-algorithm}
欲明算法之“正确”，可述其“性质”。  
例如：排序算法之输出，具如下性质——对输出列表中任意两个相异元素，居左者恒小于居右者。此即“列表有序”之形式化定义。

同理，分布式算法之正确性，亦可藉其所应满足之性质而定义。  
譬如生成锁之围栏令牌（fencing token）时（参见[“围栏阻断僵尸与延迟请求”](/zh-cn/ch9#sec_distributed_fencing_tokens)），可要求该算法满足以下三条性质：

**唯一性（Uniqueness）**  
：任两次围栏令牌请求，所获令牌值必不相同。

**单调序（Monotonic sequence）**  
：若请求 *x* 返回令牌 *t**x*，请求 *y* 返回令牌 *t**y*，且 *x* 在 *y* 启动前已完结，则必有 *t**x* < *t**y*。

**可用性（Availability）**  
：任一节点若发起围栏令牌请求且未崩溃，则终将收到响应。

一算法于某系统模型下为正确者，当且仅当其在该模型所容许之一切可能情形中，恒满足全部所要求之性质。  
然若全体节点崩溃，或网络延迟骤然趋于无穷，则任何算法皆无法完成任务。  
故欲于允诺全故障之系统模型中仍作有用保证，须明示：  
其一、保证仅适用于非最坏情形（如：至多 *f* 个节点崩溃，或延迟有界）；  
其二、区分“安全性（safety）”与“活性（liveness）”：唯一性与单调序属安全性，恒成立（无例外）；可用性属活性，仅在部分运行条件下成立；  
其三、采用部分同步模型（partially synchronous model）或失败探测器（failure detector），以在可实现前提下刻画“可预期行为”。  
此即务实之正确性边界。
#### 安全性与活性 {#sec_distributed_safety_liveness}
为明其理，宜辨两类性质：**安全性（safety）** 与 **活性（liveness）**。前例中，“唯一性”与“单调序列”属安全性性质；而“可用性”则属活性性质。

二者之别何在？一显著征兆在于：活性性质之定义常含“终将（eventually）”一词。（诚然，汝所料不差——*最终一致性（eventual consistency）* 即为一活性性质 [^115]。）

安全性常被非正式地释为“**无坏事发生**”，活性则释为“**终有好事发生**”。然此二语仅为直觉提示，不宜深究——盖“好”“坏”乃价值判断，于算法分析中既模糊又易致歧义。其精确定义实如下 [^116]：

- 若安全性性质被违反，则必可定位至某一**具体时刻**，即违规发生之瞬点（例如，若唯一性遭破，即可指出返回重复围栏令牌（fencing token）之确切操作）。一旦违规，不可撤销；损害已成定局。
- 活性性质则反之：其于某时点或未满足（例如，节点已发请求，尚未收响应），然**未来恒存满足之可能**（如终得响应）。

区分此二性质之要义，在于助吾人应对复杂系统模型。对分布式算法而言，通常要求安全性性质**恒真**——即于系统模型之一切可能情形下皆须成立 [^108]。换言之，纵使全体节点崩溃，或全网中断，算法亦不得返回错误结果（即安全性性质必须始终维系）。

然对活性性质，允设前提与例外：例如，可限定“仅当多数节点未崩溃、且网络终将从故障中恢复时”，请求方需获响应。部分同步模型（partially synchronous model）之定义即含此要件：系统**终将回归同步状态**——即任何网络中断仅持续有限时间，而后必修复。
#### 将系统模型映射至现实世界 {#mapping-system-models-to-the-real-world}
安全性与活性性质，以及系统模型，对推理分布式算法的正确性极有用。然实践实现之时，现实之芜杂复又袭来，始知系统模型仅为现实之简化抽象。

例如，崩溃-恢复模型（crash-recovery model）中的算法，通常假设稳定存储（stable storage）中之数据可幸免于崩溃。然若磁盘数据遭损坏，或因硬件故障、配置错误而被清空，当如何？  
若某服务器固件存缺陷，重启时无法识别其硬盘——纵硬盘确已正确连接于该服务器，又当如何？

法定人数算法（quorum algorithms；参见[“读写法定人数”](/zh-cn/ch6#sec_replication_quorum_condition)）依赖节点能记住其所声称已存储之数据。若节点罹患“失忆症”，遗忘此前所存数据，则法定人数条件即告失效，算法正确性亦随之瓦解。或需另立新系统模型：假定稳定存储**通常**可在崩溃中幸存，但**偶有丢失**。然此模型遂致推理难度陡增。

理论算法描述可明言：某些情形被**直接排除于考虑之外**；在非拜占庭（non-Byzantine）系统中，本就必须对可能与不可能发生的故障作出若干假设。然真实实现中，仍须编写代码以应对那些“理论上不可能发生”之事——即便该处理仅止于`printf("Sucks to be you")`与`exit(666)`，即交由人工运维者清理现场（[^119]）。  
（此即计算机科学与软件工程之一大分野。）

非谓理论抽象之系统模型全无价值——实则正相反。  
其价值正在于：将现实系统之繁复，提炼为一组有限、可控之故障类型，使吾人得以清晰把握问题本质，并系统性地寻求解法。
### 形式化方法与随机化测试 {#sec_distributed_formal}
我们何以确知一算法满足所需性质？盖因并发、局部故障与网络延迟，系统可能处于状态之数量极巨。须证诸性质于一切可能状态皆成立，并防疏漏任何边界情形。

其一，可借形式化验证：以数学语言精确定义算法，继而运用逻辑证明技术，证其在系统模型所容许之一切情形下，恒满足所求性质。然须明：算法之形式化正确，并不保证其于真实系统中之实现必无差错。然此乃极佳之始步——理论分析常能揭示算法中潜藏之缺陷，此类缺陷或于实际系统中蛰伏良久，唯待异常情境（如时序假设被打破）方暴露而出。

其二，宜将理论分析与实证测试相辅而行，以验实现是否符预期行为。诸如属性导向测试（property-based testing）、模糊测试（fuzzing）、确定性模拟测试（deterministic simulation testing, DST）等法，藉随机化手段，在广域场景中检验系统。亚马逊云服务（Amazon Web Services）等公司，已在其诸多产品中成功融合运用此类技术 [^120] [^121]。
#### 模型检验与规约语言 {#model-checking-and-specification-languages}
*模型检验器*（Model checkers）乃用于验证算法或系统行为是否符合预期之工具。算法规格须以专用语言（如 TLA+、Gallina 或 FizzBee）撰写；此类语言剥离实现细节，使人专注行为逻辑本身。模型检验器继而基于此模型，穷举所有可能事件，系统性遍历状态空间，以验证不变式（invariants）在全部可达状态中恒成立。

然模型检验实不能**严格证明**不变式对**所有可能状态**皆成立——盖因多数现实算法状态空间无穷。欲证其普适性，须诉诸形式化证明（formal proof）；此虽可行，然通常较运行模型检验为难。故模型检验器实促人将算法建模为可穷尽验证之近似版本，或施加执行边界（例如限定最大消息发送数）。若缺陷仅于超界执行中方现，则必漏检。

尽管如此，模型检验器于易用性与发现隐蔽缺陷之能力间，达致良善平衡。CockroachDB、TiDB、Kafka 等诸多分布式系统，皆借模型规格于开发早期发现并修复缺陷：[^122] [^123] [^124]。例如，研究者藉 TLA+ 揭示 Viewstamped Replication（VR）协议中，因原始文字描述含歧义，可致数据丢失之风险 [^125]。

依其设计，模型检验器**不运行真实代码**，仅执行一简化模型，该模型仅刻画协议之核心逻辑。此举使状态空间遍历可行，却隐含规格与实现脱节之风险 [^126]。虽可验证模型与真实实现行为等价，然须在真实实现中嵌入可观测性设施（instrumentation）方可施行 [^127]。
#### 故障注入 {#sec_fault_injection}
诸多缺陷，常于机器或网络故障时暴露。故障注入，乃一种有效（有时令人不安）之验证手段，用以检验系统实现于异常情形下是否仍如预期运行。其理至简：向运行中系统之环境主动注入故障，观其行为如何。

故障可为网络中断、机器宕机、磁盘损坏、进程暂停——凡计算机可能失常者，皆可模拟。

故障注入测试，通常于高度近似生产环境之环境中执行；更有甚者，径直于生产环境注入故障。Netflix 借 Chaos Monkey 工具 [^128] 推广此法。此类生产环境下的故障注入，习称“混沌工程”（Chaos Engineering），前文《可靠性与容错》一节（[“Reliability and Fault Tolerance”](/zh-cn/ch2#sec_introduction_reliability)）已述及。

执行故障注入测试前，须先部署被测系统，并同步部署故障注入协调器（coordinator）及注入脚本。协调器主司决策：何类故障、何时触发；本地或远程脚本则负责向具体节点或进程注入故障。注入脚本所用工具各异：Linux 进程可用 `kill` 命令暂停或终止；磁盘可用 `umount` 命令卸载；网络连接则可通过防火墙规则人为阻断。故障注入期间及之后，可观测系统行为，以验其是否合乎预期。

然触发各类故障所需工具繁多，致使故障注入测试编写冗繁费力。故业界常采故障注入框架（如 Jepsen）以简化流程。此类框架内置多操作系统适配，并提供大量预置故障注入器（fault injector）[^129]。  
Jepsen 在众多广泛使用的系统中屡建奇功，成功揭露诸多关键缺陷 [^130] [^131]。
#### 确定性仿真测试 {#deterministic-simulation-testing}
确定性仿真测试（DST）已成为模型检验与故障注入之外广为采用的补充手段。其状态空间探索机制与模型检验相似，但直接运行真实代码，而非抽象模型。

在 DST 中，仿真器自动执行大量随机化系统执行路径。仿真过程中，网络通信、I/O 与系统时钟等均被替换为可控模拟组件（mock），使仿真器能精确控制事件发生的顺序、时序及各类故障场景。相较人工编写的测试用例或传统故障注入，DST 可覆盖更广的边界与异常组合。若测试失败，因仿真器全程记录触发失败的确切操作序列，故可完全复现；而故障注入缺乏此类细粒度控制能力，难以稳定复现。

DST 要求仿真器能掌控所有非确定性来源（如网络延迟）。为实现代码确定性，业界通常采用以下三类策略之一：

**应用层方案**  
: 部分系统自设计之初即面向确定性执行构建。例如，DST 领域先驱之一 FoundationDB，基于异步通信库 Flow 构建；Flow 提供标准化接口，供开发者注入确定性网络仿真模块 [^132]。  
同理，TigerBeetle 是一款支持一级 DST 能力的在线事务处理（OLTP）数据库。其将系统状态建模为状态机，所有状态变更均发生于单个事件循环内；再配合确定性时钟等模拟原语，整套架构即可实现端到端确定性执行 [^133]。

**运行时层方案**  
: 具备异步运行时的语言及其主流库，常提供插入点以引入确定性。典型做法是采用单线程运行时，强制所有异步逻辑串行执行。例如，FrostDB 通过补丁方式修改 Go 运行时，令 goroutine 按序调度 [^134]；Rust 生态中的 madsim 库亦采用类似思路：它为 Tokio 异步运行时 API、AWS S3 官方 Rust SDK、Kafka Rust 客户端等提供确定性实现；应用仅需替换依赖库与运行时，无需修改业务代码，即可获得确定性测试执行能力。

**机器层方案**  
: 不在代码或运行时层面打补丁，而是令整台虚拟机具备确定性。此法极为精细，要求机器对所有原本非确定性系统调用（如 `gettimeofday`、`read`、`sendto`）返回可重现的确定性响应。Antithesis 等工具通过定制化超管理器（hypervisor）实现该目标：该 hypervisor 将时钟、网络、存储等所有非确定性操作统一重定向至确定性模拟层。完成配置后，开发者可在该 hypervisor 内以容器形式部署完整分布式系统，并获得全栈确定性执行环境。

DST 的优势不仅限于可复现性。以 Antithesis 为代表的工具，还能在测试执行中动态探测低频行为路径，并据此分叉出多个子执行流，主动探索更多代码分支。此外，因确定性测试普遍采用模拟时钟与模拟网络调用，其执行速度常远超真实挂钟时间。例如，TigerBeetle 的时间抽象机制允许仿真器在毫秒级内模拟数秒乃至数分钟的网络延迟与超时事件，无须真实等待；此类加速技术显著提升代码路径覆盖率与探索效率。
#### 确定性之力 {#sidebar_distributed_determinism}
非确定性，乃本章所论分布式系统诸挑战之核心：并发、网络延迟、进程暂停、时钟跳变、崩溃等，皆以不可预测之方式发生，且每次运行表现各异。反之，若能使系统确定化，则可极大简化其设计与验证。

事实上，确定化乃分布式系统设计中反复出现之简明而有力的思想。除确定化仿真测试外，前文已述多种应用方式：

* 事件溯源（参见［“事件溯源与CQRS”］(/zh-cn/ch3#sec_datamodels_events)）之关键优势，在于可依事件日志确定性重放，以重建派生的物化视图。  
* 工作流引擎（参见［“持久化执行与工作流”］(/zh-cn/ch5#sec_encoding_dataflow_workflows)）依赖工作流定义之确定性，方能提供持久化执行语义。  
* 状态机复制（详见［“利用共享日志”］(/zh-cn/ch10#sec_consistency_smr)），则通过在各副本上独立执行同一序列之确定性事务，实现数据复制。此前已见其二变体：基于语句的复制（参见［“复制日志的实现”］(/zh-cn/ch6#sec_replication_implementation)），及借助存储过程实现的串行事务执行（参见［“存储过程之优劣”］(/zh-cn/ch8#sec_transactions_stored_proc_tradeoffs)）。

然使代码全然确定，须审慎为之。纵已消除所有并发，并以确定性模拟替代I/O、网络通信、时钟与随机数生成器，非确定性仍或潜存。例如，某些编程语言中遍历哈希表元素之顺序即属非确定；是否触达资源上限（如内存分配失败、栈溢出），亦属非确定。
## 概要 {#summary}
本章所论，乃分布式系统中种种疑难，概有三端：

* 凡网络发包，或遗失、或任意延宕；其应答亦然。故若未得回应，实难断言消息是否已达彼端。  
* 节点时钟纵经 NTP 精调，仍常大幅偏移他节点；或骤前、或猝后；且无可靠置信区间可依，恃之则险。  
* 进程执行途中，或于任意时刻长时停顿；他节点遂判其为“死”，而其后复生，竟不自知曾遭暂停。

此类 *局部故障*（partial failures）之可发生，实为分布式系统之根本特征。凡软件涉他节点之操作，皆或偶败、或忽慢、或全无响应（终致超时）。故分布式系统之要务，在构设容错机制，使整体服务不因部分组件失效而中辍。

容错之始，在于 *故障检测*；然此亦非易事。多数系统并无精确判别节点是否宕机之法，故分布式算法多赖超时（timeout）推断远端节点是否存活。然超时无法区分网络故障与节点故障；网络延迟波动，更常致节点被误判为崩溃。至于“跛行节点”（limping nodes）——虽尚能响应，却迟缓至不可用者——其识别与处置，尤难上加难。

故障既检，容之亦非坦途：诸节点间无全局变量、无共享内存、无共同知识，亦无任何共享状态 [^83]。  
节点尚不能就“此刻为何时”达成一致，遑论更深之共识？信息唯一通路，唯藉不可靠网络传递。重大决策不可委于单点，故必赖协议协同众节点，求取法定多数（quorum）之共识。

若习于单机之理想化数学世界——同一操作恒得确定结果——则初入分布式之纷繁物理现实，不免惊愕。反之，分布式系统工程师常视单机可解之题为 trivial [^4]；今之单机，确已能承重负。故若可避开启潘多拉魔盒，仅以嵌入式存储引擎（参见[“嵌入式存储引擎”](/zh-cn/ch4#sidebar_embedded)）固守单机，则通常为上策。

然如[“分布式系统 vs 单节点系统”](/zh-cn/ch1#sec_introduction_distributed)所析，可扩展性（scalability）非采用分布式之唯一动因。容错性（fault tolerance）与低延迟（low latency，即地理邻近用户部署数据）同等关键，而此二者，单节点不可企及。分布式系统之伟力，在其原则上可永续运行、服务不中断——盖因一切故障与维护，皆可限于节点粒度之内化解。（实践中，若错误配置同步推送至全部节点，则纵为分布式，亦将倾覆。）

本章另作旁涉，探问网络、时钟与进程之不可靠性，是否自然铁律？结论否之：硬实时（hard real-time）响应保障与有界延迟（bounded delays）确可实现，然代价极高，硬件资源利用率锐减。故绝大多数非安全攸关系统，宁取廉价而不可靠，不选昂贵而高可靠。

本章专述问题，观之 bleak。下章则转述解法，详析若干专为应对分布式系统诸难而设计之算法。
